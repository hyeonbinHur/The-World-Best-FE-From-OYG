# Libraries (라이브러리 네이티브 계측)

> 원문: https://opentelemetry.io/docs/concepts/instrumentation/libraries/

---

## 한 줄 요약

라이브러리 만드는 사람이 OTel API를 직접 내장해서, 쓰는 사람이 별도 설정 없이도 텔레메트리를 얻게 하는 방법.

---

## 이 문서는 누구를 위한 건지

**라이브러리 제작자(author)** 를 위한 가이드임.

앱 개발자가 아니라, npm 패키지나 SDK를 만드는 사람이 "내 라이브러리에 OTel을 어떻게 내장하지?" 궁금할 때 보는 문서.

---

## 외부 계측 vs 네이티브 내장, 뭐가 다른지

| | 외부 Instrumentation Library | 네이티브 내장 |
|--|---------------------------|-------------|
| 누가 만듦 | OTel 커뮤니티 or 제3자 | 라이브러리 제작자 본인 |
| 품질 | 외부에서 만들다 보니 내부 구조 모를 수 있음 | 내부를 가장 잘 아는 사람이 만듦 |
| 로그-트레이스 연관 | 제한적 | 자연스러운 correlation |
| 유지보수 | 라이브러리 업데이트 시 깨질 수 있음 | 같이 업데이트 |
| 확장 포인트 | 제한적 | 문서화된 extensibility point 제공 가능 |

장점 정리:
- 기존에 있던 커스텀 로깅 훅 → OTel API로 대체 가능 (표준화)
- Trace/Log/Metric이 자연스럽게 연관됨
- Semantic Conventions 따르면 기술 스택 달라도 일관된 텔레메트리
- 사용자가 signal을 세밀하게 조절할 수 있는 확장 포인트 제공

---

## 핵심 원칙: API만 의존해라

이게 가장 중요한 규칙:

```
라이브러리 → OTel API에만 의존 (O)
라이브러리 → OTel SDK에 의존   (X) ← 절대 하지 말 것
```

왜?
- **SDK가 없으면** → API 호출이 전부 **no-op** (아무것도 안 함) → 성능 영향 제로
- **SDK가 있으면** → API 호출이 실제 텔레메트리 생성
- 라이브러리가 SDK까지 끌고 오면 → 사용자 앱과 SDK 버전 충돌, 불필요한 의존성 증가

```
사용자 앱 (SDK 선택은 여기서)
  ├── OTel SDK v1.5
  ├── 내 라이브러리 (API v1.0만 의존)
  └── 다른 라이브러리 (API v1.0만 의존)
  → 충돌 없이 잘 동작
```

### 의존성 팁
- OTel Trace API는 2021년 초에 **Stable** 됐고, SemVer 2.0 따름
- 가장 낮은 안정 버전(1.0.*)에 의존 → 호환성 극대화
- 가능하면 계측 코드를 **별도 패키지**로 분리 → 안 쓰는 사람은 설치 안 해도 됨
- OTel contrib 레포에 올리는 것도 방법 → semantic conventions 최신 유지에 유리

## Semantic Conventions (시맨틱 컨벤션) 따르기

Span에 어떤 정보를 넣을지의 **표준 규격**임.

웹 프레임워크, RPC 클라이언트, DB, 메시징 시스템별로 "이런 attribute를 이렇게 넣어라" 정해져 있음.
이걸 따르면 사용자가 라이브러리별 세부 사항을 안 배워도 됨.

**Span 이름 짓기 주의사항**:
```
좋은 예: GET /users/:id     ← 카디널리티 낮음 (패턴)
나쁜 예: GET /users/12345   ← 카디널리티 폭발 (실제 ID가 이름에 들어감)
```

- `schema_url` 속성에 사용 중인 컨벤션 버전 기록
- 컨벤션은 계속 진화 중 → 기여도 가능 (OTel Instrumentation Slack)

## 뭘 계측해야 하나

### 1. Public API — 네트워크 호출이 있는 퍼블릭 메서드

사용자가 직접 호출하는 메서드 중 네트워크 I/O나 시간 걸리는 작업이 있는 것.

```java
private Response selectWithTracing(Query query) {
    // 1. Span 생성: "SELECT {db}.{collection}" 형태로 이름
    Span span = tracer.spanBuilder(
        String.format("SELECT %s.%s", dbName, collectionName))
            .setSpanKind(SpanKind.CLIENT)          // 외부 호출이니까 CLIENT
            .setAttribute("db.name", dbName)        // semantic convention 속성
            .startSpan();

    try (Scope unused = span.makeCurrent()) {       // 2. 이 Span을 "현재 활성 Span"으로
        Response response = query.runWithRetries();
        if (response.isSuccessful()) {
            span.setStatus(StatusCode.OK);
        }
        if (span.isRecording()) {
           // 비싼 속성은 recording 체크 후 추가 (성능)
        }
    } catch (Exception e) {
        span.recordException(e);   // 3. 예외를 Span Event로 기록
        span.setStatus(StatusCode.ERROR, e.getClass().getSimpleName());
        throw e;                   // 4. 예외는 반드시 다시 던짐 (삼키면 안 됨)
    } finally {
        span.end();                // 5. 반드시 종료 (안 하면 메모리 누수)
    }
}
```

### 2. 네트워크 호출 Span — 중복 조심

OTel이 이미 HTTP/gRPC 클라이언트를 자동 계측하고 있을 수 있음.

| 상황 | 어떻게 |
|------|-------|
| OTel auto-instrumentation이 이미 있음 | 네트워크 Span 안 만들어도 됨 (중복) |
| auto-instrumentation 없는 환경도 지원해야 함 | 만들되, 비활성화 가능하게 |
| 라이브러리 고유 정보 추가하고 싶음 | 기존 Span에 attribute만 추가 |

### 3. Events vs Spans — verbose 데이터는 Event로

```
나쁜 예:
  [Span] db.query
    [Span] serialize_request    ← 이런 건 Span으로 만들면 너무 세분화
    [Span] deserialize_response

좋은 예:
  [Span] db.query
    [Event] "request serialized" at 2ms    ← Event로 충분
    [Event] "response deserialized" at 15ms
```

- 추가 Span은 트레이스를 복잡하게 만듦
- Event는 반드시 **직접 생성한 Span 인스턴스**에 attach
- Active Span에 이벤트 달지 말 것 → 내가 만든 Span이 아닐 수 있음

---

## 계측 안 해도 되는 경우

아래 조건을 **모두** 만족하면 굳이 계측 안 해도 됨:

1. 라이브러리가 thin wrapper (얇은 감싸기)일 뿐
2. OTel이 이미 하위 네트워크 호출을 계측하고 있음
3. 해당 라이브러리에 적용되는 특별한 semantic convention이 없음

> 공식 문서 원문: "When in doubt, don't instrument."
> (확신 없으면 계측하지 마라)

## Context Propagation 상세

### Extract (수신 측 — 서버, 메시지 컨슈머)

외부에서 들어온 요청의 HTTP 헤더나 메시지 메타데이터에서 trace context를 꺼내는 것.

```java
// 1. 인바운드 요청에서 Context 추출
Context extractedContext = propagator.extract(
    Context.current(), httpExchange, getter);

// 2. 추출된 Context를 부모로 Span 생성
Span span = tracer.spanBuilder("receive")
    .setSpanKind(SpanKind.SERVER)
    .setParent(extractedContext)    // ← 이래야 같은 Trace로 연결됨
    .startSpan();

// 3. Span을 current로 설정
try (Scope unused = span.makeCurrent()) {
    // 여기서부터 하위 코드의 로그, nested span이 자동으로 이 Span과 연관됨
    userCode();
} catch (Exception e) {
    span.recordException(e);
    span.setStatus(StatusCode.ERROR);
    throw e;
} finally {
    span.end();
}
```

핵심: `setParent(extractedContext)` 안 하면 새 Trace가 시작됨 → 분산 트레이싱 끊김.

### Inject (발신 측 — 클라이언트, 메시지 프로듀서)

다른 서비스를 호출할 때 현재 trace context를 HTTP 헤더에 넣는 것.

```java
Span span = tracer.spanBuilder("send")
    .setSpanKind(SpanKind.CLIENT)
    .startSpan();

try (Scope unused = span.makeCurrent()) {
    // Context를 HTTP 헤더에 주입
    propagator.inject(Context.current(), transportLayer, setter);
    send();    // 이 요청의 헤더에 traceparent가 들어감
} catch (Exception e) {
    span.recordException(e);
    span.setStatus(StatusCode.ERROR);
    throw e;
} finally {
    span.end();
}
```

### In-Process Context 전파 (비동기 처리 시 주의)

`span.makeCurrent()` 하면 같은 스레드 내에서 로그, nested span이 자동 연관됨.
**그런데 비동기/콜백/백그라운드 작업에선 이 "암묵적 전파"가 끊김.**

```
[문제 상황]
Public API 진입 → Span 생성 + makeCurrent
  └── setTimeout(() => {
        // 여기선 active span이 없음! Context가 끊김!
        doSomething();
      })
```

해결법:
1. Public API 진입 시점에 Context를 **캡처**
2. 비동기 작업에 **명시적으로 전달**
3. 전달받은 Context를 부모로 새 Span 생성

```
[해결]
Public API 진입 → context 캡처
  └── 비동기 작업에 context를 인자로 전달
        └── 전달받은 context를 부모로 Span 생성 → 연결 유지!
```

## Tracer 얻는 방법

라이브러리에서 Tracer를 어떻게 가져오느냐:

**방법 1: 의존성 주입 (추천)**
```java
// 사용자가 TracerProvider를 주입해줌
public MyLibrary(TracerProvider tracerProvider) {
    this.tracer = tracerProvider.getTracer("my-library", "1.0.0");
}
```

**방법 2: 글로벌 Provider 접근**
```java
// 글로벌에 등록된 Provider 사용
Tracer tracer = GlobalOpenTelemetry.getTracer("my-library", "1.0.0");
```

이름과 버전을 반드시 넣어야 함 → 나중에 "이 Span 누가 만든 거지?" 추적할 때 필요.

## 성능 최적화

### SDK 없으면 완전 무료

API만 의존하는 라이브러리에서 SDK가 없으면:
- 모든 API 호출이 **no-op** → 오버헤드 제로
- `tracer.startSpan()` 호출해도 아무 일도 안 일어남

### SDK 있을 때: isRecording() 체크

실제 운영에선 샘플링 때문에 대부분의 Span이 "recording 안 하는" 상태임.
비싼 연산은 체크 후 실행:

```java
Span span = tracer.spanBuilder("SELECT db.table")
    .setSpanKind(SpanKind.CLIENT)
    .setAttribute("db.name", dbName)    // 중요 속성은 생성 시 바로 세팅
    .startSpan();

if (span.isRecording()) {
    // 이 안의 코드는 실제 recording 중일 때만 실행됨
    // 쿼리 sanitize 같은 비싼 연산을 여기에
    span.setAttribute("db.statement", sanitize(query.statement()));
}
```

## 에러 처리

> OTel API는 잘못된 인자에도 실패하지 않고, 예외를 throw하지 않으며, 예외를 삼킨다.

무슨 말이냐면:
- `span.setAttribute(null, null)` 해도 앱이 안 터짐
- 계측 코드에 버그가 있어도 **애플리케이션은 정상 동작**
- 대신 계측 쪽 문제가 조용히 묻힐 수 있음 → **테스트 필수**

## 테스트 방법

Mock Exporter로 단위 테스트 가능:

```java
@Test
public void checkInstrumentation() {
    // 1. 테스트용 Exporter 생성
    SpanExporter exporter = new TestExporter();

    // 2. 테스트용 SDK 구성
    Tracer tracer = OpenTelemetrySdk.builder()
        .setTracerProvider(SdkTracerProvider.builder()
            .addSpanProcessor(SimpleSpanProcessor.create(exporter))
            .build())
        .build()
        .getTracer("test");

    // 3. 테스트 실행
    myLibrary.doSomething();

    // 4. 생성된 Span 검증
    validateSpans(exporter.exportedSpans);
}

class TestExporter implements SpanExporter {
    public final List<SpanData> exportedSpans =
        Collections.synchronizedList(new ArrayList<>());

    @Override
    public CompletableResultCode export(Collection<SpanData> spans) {
        exportedSpans.addAll(spans);
        return CompletableResultCode.ofSuccess();
    }
}
```

통합 테스트도 해야 함:
- 실제 앱에서 다른 auto-instrumentation과 같이 동작하는지
- 인바운드/아웃바운드 요청에서 Context 전파가 잘 되는지
- 로그에 Trace ID가 제대로 들어가는지

## 체크리스트

라이브러리에 OTel 내장할 때 확인 사항:

- [ ] API만 의존하고 SDK 의존 안 했는지
- [ ] Semantic Conventions 따랐는지
- [ ] Span 이름 카디널리티 적절한지
- [ ] `isRecording()` 체크로 비싼 연산 보호했는지
- [ ] 예외 발생 시 `recordException()` + `setStatus(ERROR)` + 재throw 했는지
- [ ] `span.end()` 빠짐없이 호출했는지
- [ ] Context propagation (extract/inject) 구현했는지
- [ ] 비동기 처리에서 Context 명시적 전달했는지
- [ ] OTel Registry에 등록했는지
- [ ] 테스트 작성했는지
