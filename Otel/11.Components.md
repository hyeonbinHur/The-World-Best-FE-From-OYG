# Components (OTel 구성 요소)

> 원문: https://opentelemetry.io/docs/concepts/components/

---

## 한 줄 요약

OTel은 단일 도구가 아니라, Specification + Collector + 언어별 SDK + K8s Operator + FaaS 자산으로 이뤄진 **프레임워크**임.

---

## 왜 이런 구조인지

벤더 종속(vendor lock-in)을 없애려고 만들어진 프로젝트임.

```
기존 방식:
  앱 → Datadog SDK → Datadog만 가능
  앱 → New Relic SDK → New Relic만 가능
  → 백엔드 바꾸면 SDK도 코드도 다 바꿔야 함

OTel 방식:
  앱 → OTel SDK → Collector → Datadog도, Jaeger도, Prometheus도 OK
  → 백엔드 바꿔도 앱 코드 안 건드림
```

---

## 1. Specification (명세)

모든 언어 구현이 따라야 하는 **설계도**. 세 영역:

| 영역 | 뭘 정의하나 | 쉽게 말하면 |
|------|-----------|-----------|
| **API** | 데이터 타입 + 연산. Traces/Metrics/Logs를 생성·연관시키는 인터페이스 | "Tracer, Span, Meter 이런 게 있고, 이런 메서드를 제공해야 해" |
| **SDK** | 각 언어별 API 구현 요구사항. Provider, Processor, Exporter 행동 규격 | "Java로 만들든 Python으로 만들든, 이런 동작은 보장해야 해" |
| **Data** | OTLP 프로토콜 정의 + Semantic Conventions | "데이터를 주고받는 형식, 그리고 attribute 이름 표준" |

### OTLP (OpenTelemetry Protocol)

OTel의 네이티브 데이터 전송 프로토콜. 모든 SDK가 이걸 지원함.

왜 별도 프로토콜이 필요한지:
- Jaeger 포맷, Zipkin 포맷, Prometheus 포맷 다 다름
- OTLP는 Traces/Metrics/Logs 전부 하나의 프로토콜로 통일
- **데이터 손실 없이** 원본 그대로 전송 가능 (다른 포맷은 변환 과정에서 정보 유실 가능)

## 2. Collector

> "벤더 중립 프록시. 텔레메트리 데이터를 수신하고, 처리하고, 내보낸다."

앱과 백엔드 사이에 끼어서 데이터를 중계하는 역할. **파이프라인** 구조로 동작:

```
수신(Receivers) → 처리(Processors) → 내보내기(Exporters)
```

### Receivers (수신)

다양한 포맷의 텔레메트리를 받아들임:

| Receiver | 설명 |
|----------|------|
| OTLP | OTel 네이티브 포맷 (gRPC/HTTP) |
| Jaeger | Jaeger 포맷 |
| Zipkin | Zipkin 포맷 |
| Prometheus | Prometheus scrape |
| 상용 도구 | Datadog, Splunk 등 |

→ 기존 시스템이 Jaeger 쓰고 있어도, Collector가 받아서 다른 백엔드로 보낼 수 있음.

### Processors (처리)

데이터가 Receiver에서 Exporter로 가기 전에 가공하는 단계:

| Processor | 뭐 하는 건지 |
|-----------|------------|
| **Batch** | 데이터를 모았다가 한 번에 보냄 (네트워크 효율) |
| **Filter** | 조건에 안 맞는 데이터 버림 (예: health check 요청 제외) |
| **Attributes** | attribute 추가/제거/수정 |
| **Tail Sampling** | 전체 Trace를 보고 수집 여부 결정 (에러 있는 것만 수집 등) |
| **Resource** | Resource 속성 추가/수정 |

### Exporters (내보내기)

처리된 데이터를 최종 목적지로 전송:

- 동시에 **여러 백엔드로 전송 가능** (fan-out)
- 예: 같은 데이터를 Jaeger(트레이스 분석) + Prometheus(메트릭) + S3(장기 보관)에 동시 전송

### 배포 방식

| 방식 | 설명 | 언제 쓰나 |
|------|------|----------|
| **Agent (사이드카)** | 앱과 같은 호스트/Pod에 배포 | 앱별 세밀한 설정 필요할 때 |
| **Gateway** | 별도 호스트에 독립 배포 | 중앙 집중 처리, 대규모 환경 |
| **K8s DaemonSet** | 노드당 1개씩 배포 | 쿠버네티스 환경 |
| **K8s Deployment** | 독립 Pod으로 배포 | 중앙 집중 처리 |

### 왜 Collector를 쓰나 (앱에서 직접 보내면 안 되나?)

쓰면 좋은 이유:
1. **백엔드 교체가 쉬움** — 앱 코드 안 건드리고 Collector 설정만 변경
2. **데이터 가공** — 필터링, 샘플링, 속성 추가를 앱 밖에서 처리
3. **다중 백엔드** — 같은 데이터를 여러 곳에 동시 전송
4. **버퍼링** — 백엔드가 잠깐 죽어도 Collector가 데이터를 잡고 있음
5. **보안** — 앱이 직접 외부 엔드포인트를 알 필요 없음

## 3. Language-Specific API & SDK

Go, Java, Python, JavaScript, .NET, PHP, Ruby, Rust, Erlang, Swift, C++ 등 각 언어별 구현체.

하위 구성 요소 6가지:

### 3-1. Instrumentation Libraries

인기 프레임워크/라이브러리에 대한 **자동 텔레메트리 생성기**.

```
Express.js → @opentelemetry/instrumentation-express
Spring Boot → opentelemetry-javaagent
Django → opentelemetry-instrumentation-django
```

목표: 모든 주요 라이브러리가 **"기본적으로 관측 가능(observable by default)"** 하게 만드는 것.

하는 일:
- HTTP 요청이 들어오면 자동으로 Span 생성
- DB 쿼리 실행되면 자동으로 Span + 쿼리 정보 기록
- 메시지 큐 produce/consume 시 자동으로 Context 전파

### 3-2. Exporters

텔레메트리 데이터를 백엔드가 이해하는 포맷으로 변환하고 전송하는 컴포넌트.

| Exporter | 특징 |
|----------|------|
| **OTLP** | OTel 네이티브. **데이터 손실 없음.** 가장 권장 |
| Jaeger | Jaeger 전용 포맷으로 변환 |
| Zipkin | Zipkin 전용 포맷으로 변환 |
| Prometheus | Pull 방식으로 메트릭 노출 (/metrics 엔드포인트) |
| Console/Stdout | 콘솔에 출력. 디버깅용 |
| Vendor-specific | Datadog, New Relic 등 벤더 전용 |

OTLP가 권장인 이유:
- 다른 포맷은 변환 과정에서 **정보 유실** 가능 (예: Span Links가 Jaeger 포맷에 없으면 날아감)
- OTLP는 OTel의 모든 기능을 표현할 수 있음
- 대부분의 백엔드가 OTLP 직접 수신을 지원하기 시작함

### 3-3. Zero-Code Instrumentation

소스 코드 수정 없이 API + SDK를 자동 주입하는 메커니즘.

자세한 내용은 [08-Zero-code.md](./08-Zero-code.md) 참고.

### 3-4. Resource Detectors

텔레메트리를 생성하는 **엔티티(서비스, 호스트, 컨테이너 등)의 속성**을 자동 감지해서 붙여주는 컴포넌트.

"이 텔레메트리가 어디서 나온 건지" 메타데이터를 채워줌:

| 감지 대상 | 속성 예시 | 어디서 감지 |
|----------|----------|-----------|
| **Service** | `service.name=order-service`, `service.version=1.2.0` | 환경변수/설정 |
| **Host** | `host.name=prod-server-01`, `host.arch=amd64` | OS 정보 |
| **Process** | `process.runtime.name=nodejs`, `process.pid=12345` | 런타임 정보 |
| **OS** | `os.type=linux`, `os.version=5.15` | 시스템 호출 |
| **K8s** | `k8s.pod.name=order-abc`, `k8s.namespace.name=prod` | K8s Downward API |
| **Cloud** | `cloud.provider=aws`, `cloud.region=ap-northeast-2` | 클라우드 메타데이터 API |

수동으로도 설정 가능:
```bash
OTEL_RESOURCE_ATTRIBUTES=service.name=order-service,deployment.environment=prod
```

이게 왜 중요한지:
- 대시보드에서 "이 에러가 어떤 Pod에서 났지?", "어떤 리전에서 느리지?" 필터링할 때 필요
- Resource 속성이 없으면 텔레메트리가 **출처 불명**이 됨

### 3-5. Cross-Service Propagators

서비스 간 **Context를 전달**하는 컴포넌트.

HTTP 헤더나 메시지 메타데이터에 trace ID, span ID 등을 실어 보내는 역할.

| Propagator | 포맷 | 비고 |
|-----------|------|------|
| **W3C TraceContext** | `traceparent` 헤더 | OTel 기본값. 표준 |
| **W3C Baggage** | `baggage` 헤더 | key-value 데이터 전파 |
| B3 | `X-B3-TraceId` 등 | Zipkin 호환. 레거시 시스템 |
| Jaeger | `uber-trace-id` 헤더 | Jaeger 레거시 |

여러 Propagator를 동시에 쓸 수 있음 (마이그레이션 시 유용):
```bash
OTEL_PROPAGATORS=tracecontext,baggage,b3
```

### 3-6. Samplers

생성되는 트레이스의 **양을 조절**하는 컴포넌트. 비용과 성능 관리 목적.

모든 요청을 100% 수집하면:
- 스토리지 비용 폭발
- 네트워크 부하
- 백엔드 처리 지연

그래서 일부만 수집하는 **샘플링** 필요.

**Head Sampling** (트레이스 시작 시점에 수집 여부 결정):

| 전략 | 동작 | 언제 쓰나 |
|------|------|----------|
| **AlwaysOn** | 100% 수집 | 개발/테스트 환경 |
| **AlwaysOff** | 0% 수집 (전부 버림) | 계측은 해두되 수집은 안 할 때 |
| **TraceIdRatioBased** | 비율 기반 (예: 10%만 수집) | 프로덕션. 트래픽 많을 때 |
| **ParentBased** | 부모 Span의 샘플링 결정 따름 | 분산 환경에서 일관성 유지 |

참고: **Tail Sampling**은 Collector에서 처리. 전체 Trace를 다 보고 나서 수집 여부를 결정함.
(예: "에러가 있는 Trace만 수집", "3초 이상 걸린 Trace만 수집")

## 4. Kubernetes Operator

OTel의 **K8s 전용 관리 도구**. 두 가지 역할:

### Collector 배포 관리

CRD(Custom Resource Definition)로 Collector 인스턴스를 선언적으로 관리:

```yaml
apiVersion: opentelemetry.io/v1alpha1
kind: OpenTelemetryCollector
metadata:
  name: otel-collector
spec:
  mode: deployment    # 또는 daemonset, sidecar
  config: |
    receivers:
      otlp:
        protocols:
          grpc:
          http:
    processors:
      batch:
    exporters:
      otlp:
        endpoint: "jaeger:4317"
    service:
      pipelines:
        traces:
          receivers: [otlp]
          processors: [batch]
          exporters: [otlp]
```

### 워크로드 자동 계측

Pod에 annotation만 달면 auto-instrumentation 자동 주입:

```yaml
metadata:
  annotations:
    instrumentation.opentelemetry.io/inject-java: "true"
    # 또는 inject-python, inject-nodejs, inject-dotnet
```

이러면 Pod 시작 시 init container가 OTel Agent를 자동 주입.
앱 이미지 수정 없이 계측 가능.

## 5. Function as a Service (FaaS) Assets

**서버리스 환경**(AWS Lambda 등)에서 OTel을 쓰기 위한 사전 구성 패키지.

| 유형 | 뭘 하는 건지 |
|------|------------|
| **Auto-instrumentation Lambda Layer** | Lambda 함수에 OTel Agent 자동 주입. 코드 수정 없이 트레이싱 |
| **Collector Lambda Layer** | Lambda 환경에서 Collector를 사이드카처럼 실행. 텔레메트리 수집·전송 |

서버리스 환경의 특수성:
- 콜드 스타트 시 Collector도 같이 초기화돼야 함
- 함수 종료 시 버퍼에 남은 텔레메트리를 flush해야 함
- Lambda Layer로 이 문제를 해결

## 구성 요소 선택 가이드

| 상황 | 필요한 것 |
|------|----------|
| 앱에서 텔레메트리 만들고 싶다 | **Language SDK** (API + SDK) |
| 코드 수정 없이 자동 계측하고 싶다 | **Zero-Code Instrumentation** |
| 텔레메트리를 백엔드로 보내고 싶다 | **Exporter** (직접 전송) 또는 **Collector** (중계) |
| 여러 백엔드에 동시 전송, 필터링, 가공하고 싶다 | **Collector** |
| K8s에서 Collector/계측 관리하고 싶다 | **K8s Operator** |
| Lambda에서 트레이싱하고 싶다 | **FaaS Assets** |
