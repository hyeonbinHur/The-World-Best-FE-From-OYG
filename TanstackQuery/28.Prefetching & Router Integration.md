# Prefetching & Router Integration (프리페칭 및 라우터 통합)

특정 데이터가 필요할 것으로 예상되거나 확실할 때, 프리페칭을 사용하여 미리 캐시에 데이터를 채워 더 빠른 사용자 경험을 제공할 수 있습니다.

## 프리페칭 패턴

프리페칭에는 몇 가지 다른 패턴이 있습니다:

1. **이벤트 핸들러에서**
2. **컴포넌트에서**
3. **라우터 통합을 통해**
4. **서버 렌더링 중에** (라우터 통합의 또 다른 형태)

이 가이드에서는 처음 세 가지를 살펴보고, 네 번째는 Server Rendering & Hydration 가이드와 Advanced Server Rendering 가이드에서 자세히 다룹니다.

프리페칭의 구체적인 사용 사례 중 하나는 Request Waterfalls를 피하는 것입니다. 이에 대한 심층적인 배경과 설명은 Performance & Request Waterfalls 가이드를 참조하세요.

## prefetchQuery & prefetchInfiniteQuery

다양한 프리페치 패턴을 살펴보기 전에, `prefetchQuery`와 `prefetchInfiniteQuery` 함수를 먼저 알아보겠습니다.

### 기본 사항

- 기본적으로 이 함수들은 `queryClient`에 설정된 기본 `staleTime`을 사용하여 캐시의 기존 데이터가 신선한지 또는 다시 가져와야 하는지를 결정합니다
- 특정 `staleTime`을 전달할 수도 있습니다: `prefetchQuery({ queryKey: ['todos'], queryFn: fn, staleTime: 5000 })`
- 이 `staleTime`은 프리페치에만 사용되며, `useQuery` 호출에도 별도로 설정해야 합니다
- `staleTime`을 무시하고 캐시에 데이터가 있으면 항상 반환하려면 `ensureQueryData` 함수를 사용할 수 있습니다

**팁**: 서버에서 프리페칭하는 경우, 각 프리페치 호출에 특정 `staleTime`을 전달하지 않으려면 해당 `queryClient`에 0보다 높은 기본 `staleTime`을 설정하세요.

- 프리페치된 쿼리에 대한 `useQuery` 인스턴스가 없으면 `gcTime`에 지정된 시간 후 삭제되고 가비지 컬렉션됩니다
- 이 함수들은 `Promise<void>`를 반환하므로 쿼리 데이터를 반환하지 않습니다. 필요한 경우 `fetchQuery`/`fetchInfiniteQuery`를 대신 사용하세요
- 프리페치 함수는 오류를 발생시키지 않습니다. 왜냐하면 일반적으로 `useQuery`에서 다시 가져오려고 시도하기 때문입니다 (우아한 폴백). 오류를 캐치해야 하는 경우 `fetchQuery`/`fetchInfiniteQuery`를 사용하세요

### 사용 예시

```tsx
const prefetchTodos = async () => {
  // 이 쿼리의 결과는 일반 쿼리처럼 캐시됩니다
  await queryClient.prefetchQuery({
    queryKey: ['todos'],
    queryFn: fetchTodos,
  })
}
```

### Infinite Queries 프리페칭

Infinite Queries는 일반 쿼리처럼 프리페치할 수 있습니다. 기본적으로 쿼리의 첫 번째 페이지만 프리페치되어 주어진 QueryKey 아래에 저장됩니다. 둘 이상의 페이지를 프리페치하려면 `pages` 옵션을 사용할 수 있으며, 이 경우 `getNextPageParam` 함수도 제공해야 합니다:

```tsx
const prefetchProjects = async () => {
  // 이 쿼리의 결과는 일반 쿼리처럼 캐시됩니다
  await queryClient.prefetchInfiniteQuery({
    queryKey: ['projects'],
    queryFn: fetchProjects,
    initialPageParam: 0,
    getNextPageParam: (lastPage, pages) => lastPage.nextCursor,
    pages: 3, // 처음 3페이지 프리페치
  })
}
```

## 이벤트 핸들러에서 프리페치

간단한 프리페칭 방법은 사용자가 무언가와 상호작용할 때 수행하는 것입니다. 이 예시에서는 `onMouseEnter` 또는 `onFocus`에서 `queryClient.prefetchQuery`를 사용하여 프리페치를 시작합니다.

```tsx
function ShowDetailsButton() {
  const queryClient = useQueryClient()

  const prefetch = () => {
    queryClient.prefetchQuery({
      queryKey: ['details'],
      queryFn: getDetailsData,
      // 프리페치는 데이터가 staleTime보다 오래된 경우에만 실행되므로,
      // 이런 경우에는 반드시 설정해야 합니다
      staleTime: 60000,
    })
  }

  return (
    <button onMouseEnter={prefetch} onFocus={prefetch} onClick={...}>
      Show Details
    </button>
  )
}
```

## 컴포넌트에서 프리페치

컴포넌트 라이프사이클 중 프리페칭은 자식이나 하위 컴포넌트가 특정 데이터를 필요로 할 것을 알지만, 다른 쿼리가 로딩을 완료하기 전까지는 렌더링할 수 없을 때 유용합니다.

### 기본 예시

```tsx
function Article({ id }) {
  const { data: articleData, isPending } = useQuery({
    queryKey: ['article', id],
    queryFn: getArticleById,
  })

  if (isPending) {
    return 'Loading article...'
  }

  return (
    <>
      <ArticleHeader articleData={articleData} />
      <ArticleBody articleData={articleData} />
      <Comments id={id} />
    </>
  )
}

function Comments({ id }) {
  const { data, isPending } = useQuery({
    queryKey: ['article-comments', id],
    queryFn: getArticleCommentsById,
  })

  ...
}
```

이는 다음과 같은 요청 워터폴을 발생시킵니다:

```plaintext
1. |> getArticleById()
2.   |> getArticleCommentsById()
```

### 해결 방법 1: 쿼리를 사용하되 결과 무시

```tsx
function Article({ id }) {
  const { data: articleData, isPending } = useQuery({
    queryKey: ['article', id],
    queryFn: getArticleById,
  })

  // 프리페치
  useQuery({
    queryKey: ['article-comments', id],
    queryFn: getArticleCommentsById,
    // 선택적 최적화: 이 쿼리가 변경될 때 리렌더링 방지
    notifyOnChangeProps: [],
  })

  if (isPending) {
    return 'Loading article...'
  }

  return (
    <>
      <ArticleHeader articleData={articleData} />
      <ArticleBody articleData={articleData} />
      <Comments id={id} />
    </>
  )
}
```

이렇게 하면 'article-comments'를 즉시 가져와 워터폴을 평탄화합니다:

```plaintext
1. |> getArticleById()
1. |> getArticleCommentsById()
```

### Suspense와 함께 프리페칭

Suspense와 함께 프리페칭하려면 조금 다르게 해야 합니다. `useSuspenseQueries`를 프리페치에 사용할 수 없습니다 (프리페치가 컴포넌트 렌더링을 차단하기 때문입니다). 또한 `useQuery`를 프리페치에 사용할 수도 없습니다 (suspenseful 쿼리가 해결될 때까지 프리페치가 시작되지 않기 때문입니다).

이 시나리오에서는 라이브러리에서 제공하는 `usePrefetchQuery` 또는 `usePrefetchInfiniteQuery` 훅을 사용할 수 있습니다.

```tsx
function ArticleLayout({ id }) {
  usePrefetchQuery({
    queryKey: ['article-comments', id],
    queryFn: getArticleCommentsById,
  })

  return (
    <Suspense fallback="Loading article">
      <Article id={id} />
    </Suspense>
  )
}

function Article({ id }) {
  const { data: articleData, isPending } = useSuspenseQuery({
    queryKey: ['article', id],
    queryFn: getArticleById,
  })

  ...
}
```

### 해결 방법 2: 쿼리 함수 내부에서 프리페치

article을 가져올 때마다 comments도 필요할 가능성이 높다는 것을 알고 있다면 이 방법이 적합합니다. 이를 위해 `queryClient.prefetchQuery`를 사용합니다:

```tsx
const queryClient = useQueryClient()
const { data: articleData, isPending } = useQuery({
  queryKey: ['article', id],
  queryFn: (...args) => {
    queryClient.prefetchQuery({
      queryKey: ['article-comments', id],
      queryFn: getArticleCommentsById,
    })

    return getArticleById(...args)
  },
})
```

### 해결 방법 3: Effect에서 프리페치

Effect에서 프리페칭하는 것도 작동하지만, 같은 컴포넌트에서 `useSuspenseQuery`를 사용하는 경우 이 effect는 쿼리가 완료된 후에 실행되므로 원하는 대로 작동하지 않을 수 있습니다.

```tsx
const queryClient = useQueryClient()

useEffect(() => {
  queryClient.prefetchQuery({
    queryKey: ['article-comments', id],
    queryFn: getArticleCommentsById,
  })
}, [queryClient, id])
```

### 요약

컴포넌트 라이프사이클 중 쿼리를 프리페치하려면 다음 방법 중 상황에 가장 적합한 것을 선택하세요:

1. Suspense 경계 전에 `usePrefetchQuery` 또는 `usePrefetchInfiniteQuery` 훅 사용
2. `useQuery` 또는 `useSuspenseQueries` 사용하고 결과 무시
3. 쿼리 함수 내부에서 프리페치
4. Effect에서 프리페치

## Dependent Queries & Code Splitting

때로는 다른 fetch의 결과를 기반으로 조건부로 프리페치하고 싶을 때가 있습니다:

```tsx
// GraphFeedItem 컴포넌트를 지연 로드합니다.
// 무언가가 렌더링할 때까지 로드를 시작하지 않습니다
const GraphFeedItem = React.lazy(() => import('./GraphFeedItem'))

function Feed() {
  const { data, isPending } = useQuery({
    queryKey: ['feed'],
    queryFn: getFeed,
  })

  if (isPending) {
    return 'Loading feed...'
  }

  return (
    <>
      {data.map((feedItem) => {
        if (feedItem.type === 'GRAPH') {
          return <GraphFeedItem key={feedItem.id} feedItem={feedItem} />
        }

        return <StandardFeedItem key={feedItem.id} feedItem={feedItem} />
      })}
    </>
  )
}

// GraphFeedItem.tsx
function GraphFeedItem({ feedItem }) {
  const { data, isPending } = useQuery({
    queryKey: ['graph', feedItem.id],
    queryFn: getGraphDataById,
  })

  ...
}
```

이 예시는 다음과 같은 이중 요청 워터폴을 발생시킵니다:

```plaintext
1. |> getFeed()
2.   |> JS for <GraphFeedItem>
3.     |> getGraphDataById()
```

조건부 프리페칭을 활용하여 최소한 코드와 데이터를 병렬로 로드할 수 있습니다:

```tsx
function Feed() {
  const queryClient = useQueryClient()
  const { data, isPending } = useQuery({
    queryKey: ['feed'],
    queryFn: async (...args) => {
      const feed = await getFeed(...args)

      for (const feedItem of feed) {
        if (feedItem.type === 'GRAPH') {
          queryClient.prefetchQuery({
            queryKey: ['graph', feedItem.id],
            queryFn: getGraphDataById,
          })
        }
      }

      return feed
    }
  })

  ...
}
```

이렇게 하면 코드와 데이터가 병렬로 로드됩니다:

```plaintext
1. |> getFeed()
2.   |> JS for <GraphFeedItem>
2.   |> getGraphDataById()
```

**트레이드오프**: `getGraphDataById`의 코드가 이제 `JS for <GraphFeedItem>` 대신 부모 번들에 포함됩니다. GraphFeedItem이 자주 발생할 가능성이 있다면 부모에 코드를 포함하는 것이 좋습니다. 매우 드문 경우라면 그렇지 않을 수 있습니다.

## 라우터 통합

컴포넌트 트리 자체에서 데이터 페칭을 수행하면 쉽게 요청 워터폴이 발생할 수 있고, 이에 대한 다양한 수정 사항들이 애플리케이션 전체에 누적되면 번거로울 수 있습니다. 매력적인 프리페칭 방법은 라우터 수준에서 통합하는 것입니다.

이 접근 방식에서는 각 라우트에 대해 해당 컴포넌트 트리에 필요한 데이터를 미리 명시적으로 선언합니다. Server Rendering은 전통적으로 렌더링이 시작되기 전에 모든 데이터를 로드해야 했기 때문에, 이것은 오랫동안 SSR 앱의 지배적인 접근 방식이었습니다.

### TanStack Router 예시

TanStack Router와 함께 작동하는 예시를 살펴보겠습니다. 전체 React Query 예시는 [TanStack Router 문서](https://tanstack.com/router)에서 확인할 수 있습니다.

라우터 수준에서 통합할 때 다음을 선택할 수 있습니다:

1. **모든 데이터가 준비될 때까지 라우트 렌더링 차단**
2. **프리페치 시작하되 결과를 기다리지 않음** (가능한 빨리 라우트 렌더링)
3. **혼합 접근**: 일부 중요한 데이터는 기다리고, 보조 데이터는 로딩이 완료되기 전에 렌더링 시작

이 예시에서는 `/article` 라우트를 article 데이터 로딩이 완료될 때까지 렌더링하지 않도록 구성하고, comments는 가능한 빨리 프리페치를 시작하지만 comments 로딩이 완료되지 않았어도 라우트 렌더링은 차단하지 않습니다.

```tsx
const queryClient = new QueryClient()
const routerContext = new RouterContext()
const rootRoute = routerContext.createRootRoute({
  component: () => { ... }
})

const articleRoute = new Route({
  getParentRoute: () => rootRoute,
  path: 'article',
  beforeLoad: () => {
    return {
      articleQueryOptions: { queryKey: ['article'], queryFn: fetchArticle },
      commentsQueryOptions: { queryKey: ['comments'], queryFn: fetchComments },
    }
  },
  loader: async ({
    context: { queryClient },
    routeContext: { articleQueryOptions, commentsQueryOptions },
  }) => {
    // comments를 가능한 빨리 가져오되, 차단하지 않음
    queryClient.prefetchQuery(commentsQueryOptions)

    // article을 가져올 때까지 라우트를 전혀 렌더링하지 않음
    await queryClient.prefetchQuery(articleQueryOptions)
  },
  component: ({ useRouteContext }) => {
    const { articleQueryOptions, commentsQueryOptions } = useRouteContext()
    const articleQuery = useQuery(articleQueryOptions)
    const commentsQuery = useQuery(commentsQueryOptions)

    return (
      ...
    )
  },
  errorComponent: () => 'Oh crap!',
})
```

다른 라우터와의 통합도 가능합니다. [react-router 예시](https://tanstack.com/query/latest/docs/framework/react/examples/react-router)를 참조하세요.

## 쿼리 수동 프라이밍

쿼리에 대한 데이터를 이미 동기적으로 사용할 수 있다면 프리페치할 필요가 없습니다. Query Client의 `setQueryData` 메서드를 사용하여 키별로 쿼리의 캐시된 결과를 직접 추가하거나 업데이트할 수 있습니다.

```tsx
queryClient.setQueryData(['todos'], todos)
```

## 추가 읽기

fetch하기 전에 Query Cache에 데이터를 가져오는 방법에 대한 심층 분석은 TkDodo의 [Seeding the Query Cache](https://tkdodo.eu/blog/seeding-the-query-cache) 문서를 참조하세요.

서버 사이드 라우터 및 프레임워크와의 통합은 방금 본 것과 매우 유사하며, 데이터를 서버에서 클라이언트로 전달하여 거기에서 캐시에 hydrate해야 한다는 점이 추가됩니다. 자세한 내용은 Server Rendering & Hydration 가이드를 참조하세요.
