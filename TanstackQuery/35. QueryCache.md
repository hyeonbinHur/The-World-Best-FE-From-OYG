# QueryCache

## 핵심 요약

QueryCache는 TanStack Query의 **저장소 역할**을 하는 핵심 메커니즘입니다. 모든 쿼리의 데이터, 메타 정보, 상태를 관리하며, 일반적으로 직접 사용하기보다는 QueryClient를 통해 간접적으로 사용합니다.

## QueryCache란?

QueryCache는 TanStack Query의 저장 메커니즘입니다. 포함된 쿼리들의 모든 데이터, 메타 정보, 상태를 저장합니다.

일반적으로 QueryCache와 직접 상호작용하지 않고, 특정 캐시를 위한 QueryClient를 사용합니다.

## 기본 사용법

```tsx
import { QueryCache } from '@tanstack/react-query'

const queryCache = new QueryCache({
  onError: (error) => {
    console.log(error)
  },
  onSuccess: (data) => {
    console.log(data)
  },
  onSettled: (data, error) => {
    console.log(data, error)
  },
})

const query = queryCache.find(['posts'])
```

## 주요 메서드

### 1. `queryCache.find`
- **역할**: 캐시에서 기존 쿼리 인스턴스를 가져오는 동기 메서드
- **반환값**: 쿼리가 존재하면 Query 인스턴스, 없으면 `undefined`
- **용도**: 쿼리의 상세 정보가 필요한 특수한 경우 (예: `query.state.dataUpdatedAt` 타임스탬프를 확인하여 쿼리가 초기값으로 사용하기에 충분히 신선한지 판단)

```tsx
const query = queryCache.find(queryKey)
```

**옵션**:
- `filters?: QueryFilters` - 쿼리 필터

### 2. `queryCache.findAll`
- **역할**: 쿼리 키와 부분적으로 일치하는 여러 쿼리 인스턴스를 가져오는 고급 동기 메서드
- **반환값**: 쿼리 배열 (없으면 빈 배열)
- **용도**: 여러 쿼리에 대한 정보가 필요한 특수한 경우

```tsx
const queries = queryCache.findAll(queryKey)
```

**옵션**:
- `queryKey?: QueryKey` - 쿼리 키
- `filters?: QueryFilters` - 쿼리 필터

### 3. `queryCache.subscribe`
- **역할**: 쿼리 캐시 전체를 구독하여 안전한/알려진 업데이트를 감지
- **감지 대상**: 쿼리 상태 변경, 쿼리 업데이트/추가/제거
- **반환값**: `unsubscribe` 함수

```tsx
const callback = (event) => {
  console.log(event.type, event.query)
}

const unsubscribe = queryCache.subscribe(callback)
```

**옵션**:
- `callback: (event: QueryCacheNotifyEvent) => void` - 캐시가 추적된 업데이트 메커니즘을 통해 업데이트될 때마다 호출되는 함수 (예: `query.setState`, `queryClient.removeQueries` 등)

**주의**: 범위를 벗어난 캐시 변경은 권장되지 않으며 구독 콜백을 실행하지 않습니다.

### 4. `queryCache.clear`
- **역할**: 캐시를 완전히 비우고 새로 시작
- **용도**: 전체 캐시 초기화

```tsx
queryCache.clear()
```

## QueryCache 옵션

### `onError?: (error: unknown, query: Query) => void`
- **선택사항**
- 일부 쿼리에서 에러가 발생하면 호출됩니다

### `onSuccess?: (data: unknown, query: Query) => void`
- **선택사항**
- 일부 쿼리가 성공하면 호출됩니다

### `onSettled?: (data: unknown | undefined, error: unknown | null, query: Query) => void`
- **선택사항**
- 일부 쿼리가 완료되면(성공 또는 에러) 호출됩니다

## 어려운 개념 풀이

### 1. "추적된 업데이트 메커니즘 (tracked update mechanisms)"
QueryCache는 내부적으로 변경 사항을 추적합니다. `query.setState`, `queryClient.removeQueries` 같은 공식 API를 통한 변경만 구독 콜백을 트리거합니다. 직접 캐시 객체를 조작하는 것은 권장되지 않으며, 이런 경우 구독 이벤트가 발생하지 않습니다.

### 2. "동기 메서드 (synchronous method)"
`find`와 `findAll`은 동기적으로 즉시 결과를 반환합니다. 비동기 작업을 기다리지 않으므로, 현재 캐시 상태의 스냅샷을 즉시 가져옵니다.

### 3. 언제 QueryCache를 직접 사용할까?
대부분의 애플리케이션에서는 QueryClient만 사용하면 충분합니다. QueryCache를 직접 사용하는 경우는:
- 쿼리의 내부 상태를 깊게 검사해야 할 때
- 캐시 전체의 변경 사항을 실시간으로 모니터링해야 할 때
- 여러 쿼리의 타임스탬프나 상태를 비교해야 할 때

### 4. QueryClient vs QueryCache
- **QueryClient**: 쿼리와 뮤테이션을 관리하는 상위 레벨 인터페이스 (일반적으로 이것을 사용)
- **QueryCache**: 실제 데이터를 저장하는 저수준 저장소 (특수한 경우에만 직접 접근)

## 요점 정리

1. **QueryCache는 저장소**: 모든 쿼리 데이터와 상태를 보관
2. **직접 사용은 드물다**: 대부분 QueryClient를 통해 간접 사용
3. **4가지 주요 메서드**: find (단일 쿼리 검색), findAll (복수 쿼리 검색), subscribe (변경 감지), clear (전체 초기화)
4. **3가지 콜백 옵션**: onError, onSuccess, onSettled로 쿼리 이벤트 처리
5. **고급 기능**: 쿼리 인스턴스의 내부 상태 접근이 필요한 특수한 경우에 유용
