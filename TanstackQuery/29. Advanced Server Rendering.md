# 고급 서버 렌더링 (Advanced Server Rendering)

## 개요

React Query를 스트리밍, 서버 컴포넌트, Next.js app router와 함께 사용하는 방법을 다루는 가이드입니다.

## 핵심 개념

### Server Components와 Next.js App Router

- **Server Components**: 서버에서만 실행되는 컴포넌트
  - 초기 페이지 뷰와 페이지 전환 시 모두 서버에서만 실행됨
  - Server Components를 "프레임워크 로더"처럼 생각하면 이해하기 쉬움

### 용어 정리

- **Server Components**: 서버에서만 실행되는 것이 보장됨
- **Client Components**: 서버와 클라이언트 양쪽에서 실행될 수 있음
  - SSR 중에는 서버에서도 렌더링됨
  - Server Components는 "로더 단계"(항상 서버), Client Components는 "애플리케이션 단계"(서버 또는 브라우저)에서 실행

## 초기 설정

### QueryClient Provider 설정

```tsx
// app/providers.tsx
'use client'

import {
  isServer,
  QueryClient,
  QueryClientProvider,
} from '@tanstack/react-query'

function makeQueryClient() {
  return new QueryClient({
    defaultOptions: {
      queries: {
        // SSR에서는 클라이언트에서 즉시 재요청하는 것을 방지하기 위해
        // staleTime을 0보다 크게 설정
        staleTime: 60 * 1000,
      },
    },
  })
}

let browserQueryClient: QueryClient | undefined = undefined

function getQueryClient() {
  if (isServer) {
    // 서버: 항상 새로운 query client 생성
    return makeQueryClient()
  } else {
    // 브라우저: 이미 있으면 재사용, 없으면 새로 생성
    // React가 초기 렌더링 중 suspend할 때 클라이언트를 버리지 않도록 하기 위해 중요
    if (!browserQueryClient) browserQueryClient = makeQueryClient()
    return browserQueryClient
  }
}

export default function Providers({ children }: { children: React.ReactNode }) {
  const queryClient = getQueryClient()

  return (
    <QueryClientProvider client={queryClient}>{children}</QueryClientProvider>
  )
}
```

### Layout 설정

```tsx
// app/layout.tsx
import Providers from './providers'

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en">
      <body>
        <Providers>{children}</Providers>
      </body>
    </html>
  )
}
```

## 데이터 Prefetching 및 De/Hydration

### Pages Router 방식 (참고용)

```tsx
// pages/posts.tsx
export async function getStaticProps() {
  const queryClient = new QueryClient()

  await queryClient.prefetchQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
  })

  return {
    props: {
      dehydratedState: dehydrate(queryClient),
    },
  }
}
```

### App Router 방식 (Server Component)

```tsx
// app/posts/page.tsx
import { dehydrate, HydrationBoundary, QueryClient } from '@tanstack/react-query'
import Posts from './posts'

export default async function PostsPage() {
  const queryClient = new QueryClient()

  await queryClient.prefetchQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
  })

  return (
    // props로 직렬화만 하면 됨
    <HydrationBoundary state={dehydrate(queryClient)}>
      <Posts />
    </HydrationBoundary>
  )
}
```

### Client Component

```tsx
// app/posts/posts.tsx
'use client'

export default function Posts() {
  // 어디서든 useQuery 사용 가능, 데이터는 즉시 사용 가능
  const { data } = useQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
  })

  // 서버에서 prefetch하지 않은 쿼리는 클라이언트에서 fetch 시작
  const { data: commentsData } = useQuery({
    queryKey: ['posts-comments'],
    queryFn: getComments,
  })

  // ...
}
```

**중요**: 
- 파일명만 Next.js 특화, 나머지는 다른 프레임워크에서도 동일
- TypeScript 5.1.3 미만 또는 @types/react 18.2.8 미만에서는 타입 에러 발생 가능
- queryFn에 함수 참조를 전달하면 직렬화 에러 발생 가능 (함수를 직접 호출할 것)

## Server Components 중첩

여러 레벨에서 Server Components를 중첩하여 데이터를 사용 위치에 가깝게 prefetch 가능:

```tsx
// app/posts/page.tsx
export default async function PostsPage() {
  const queryClient = new QueryClient()

  await queryClient.prefetchQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
  })

  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <Posts />
      <CommentsServerComponent />
    </HydrationBoundary>
  )
}

// app/posts/comments-server.tsx
export default async function CommentsServerComponent() {
  const queryClient = new QueryClient()

  await queryClient.prefetchQuery({
    queryKey: ['posts-comments'],
    queryFn: getComments,
  })

  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <Comments />
    </HydrationBoundary>
  )
}
```

**주의**: 이 패턴은 서버 사이드 waterfall을 발생시킴:
```
1. |> getPosts()
2.   |> getComments()
```

Next.js에서는 parallel routes를 사용하면 자동으로 병렬 처리됨.

## 대안: 단일 QueryClient 재사용

```tsx
// app/getQueryClient.tsx
import { QueryClient } from '@tanstack/react-query'
import { cache } from 'react'

// cache()는 요청별로 범위가 지정되어 요청 간 데이터 누수 방지
const getQueryClient = cache(() => new QueryClient())
export default getQueryClient
```

**장점**: 유틸리티 함수를 포함한 어디서든 getQueryClient() 호출 가능

**단점**: dehydrate() 호출 시마다 전체 queryClient를 직렬화 (불필요한 오버헤드)

## 데이터 소유권과 재검증

### 문제점

```tsx
export default async function PostsPage() {
  const queryClient = new QueryClient()

  // fetchQuery() 사용
  const posts = await queryClient.fetchQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
  })

  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      {/* Server Component에서 데이터 렌더링 */}
      <div>Nr of posts: {posts.length}</div>
      {/* Client Component에서도 같은 데이터 사용 */}
      <Posts />
    </HydrationBoundary>
  )
}
```

**문제**: 클라이언트에서 쿼리가 재검증되면 Server Component 부분({posts.length})은 업데이트되지 않아 동기화 문제 발생

### 권장 사항

- **staleTime: Infinity**로 설정하지 않는 한, Server Component와 Client Component 간 데이터 소유권 분리
- Server Components는 **prefetch 목적으로만** 사용
- queryClient.fetchQuery()는 에러 처리가 필요한 경우에만 사용
- fetchQuery() 결과를 Server Component에서 렌더링하거나 다른 컴포넌트에 전달하지 말 것

### React Query를 Server Components와 함께 사용하기 좋은 경우

1. React Query를 사용하는 앱을 Server Components로 마이그레이션할 때
2. 익숙한 프로그래밍 패러다임을 유지하면서 Server Components의 이점을 활용하고 싶을 때
3. 프레임워크가 제공하지 않는 React Query의 특정 기능이 필요할 때

## Server Components와 스트리밍

### 기본 스트리밍

Next.js app router는 `<Suspense>` 경계를 따라 자동으로 준비된 콘텐츠를 브라우저에 스트리밍합니다. loading.tsx 파일 생성 시 자동으로 `<Suspense>` 경계가 생성됩니다.

### Pending Queries 스트리밍 (v5.40.0+)

prefetch를 await하지 않고도 pending 쿼리를 dehydrate하여 클라이언트로 전송 가능:

```tsx
// app/get-query-client.ts
import {
  isServer,
  QueryClient,
  defaultShouldDehydrateQuery,
} from '@tanstack/react-query'

function makeQueryClient() {
  return new QueryClient({
    defaultOptions: {
      queries: {
        staleTime: 60 * 1000,
      },
      dehydrate: {
        // pending 쿼리도 dehydration에 포함
        shouldDehydrateQuery: (query) =>
          defaultShouldDehydrateQuery(query) ||
          query.state.status === 'pending',
        shouldRedactErrors: (error) => {
          // Next.js 서버 에러는 redact하지 않음
          // (Next.js가 동적 페이지를 감지하는 방식)
          return false
        },
      },
    },
  })
}
```

### await 없이 Prefetch

```tsx
// app/posts/page.tsx
export default function PostsPage() {
  const queryClient = getQueryClient()

  // await 없이 prefetch
  queryClient.prefetchQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
  })

  return (
    <HydrationBoundary state={dehydrate(queryClient)}>
      <Posts />
    </HydrationBoundary>
  )
}

// app/posts/posts.tsx
'use client'

export default function Posts() {
  // useSuspenseQuery로 Promise 사용
  const { data } = useSuspenseQuery({ 
    queryKey: ['posts'], 
    queryFn: getPosts 
  })

  // ...
}
```

**참고**: useQuery도 사용 가능하지만, Next.js가 suspend하지 않아 pending 상태로 렌더링되고 서버 렌더링이 비활성화됨.

### 비-JSON 데이터 타입 직렬화

```tsx
// app/get-query-client.ts
function makeQueryClient() {
  return new QueryClient({
    defaultOptions: {
      hydrate: {
        deserializeData: deserialize,
      },
      dehydrate: {
        serializeData: serialize,
      },
    },
  })
}

// app/posts/page.tsx
export default function PostsPage() {
  const queryClient = getQueryClient()

  queryClient.prefetchQuery({
    queryKey: ['posts'],
    queryFn: () => getPosts().then(serialize), // 서버에서 직렬화
  })

  // ...
}
```

Temporal datetime 객체 등 비-JSON 타입도 transformer를 통해 직렬화/역직렬화 가능.

## Persist Adapter와 스트리밍

Persist adapter 사용 시 pending 쿼리를 스토리지에 저장하지 않도록 주의:

```tsx
<PersistQueryClientProvider
  client={queryClient}
  persistOptions={{
    persister,
    // 성공한 쿼리만 저장 (Promise는 저장 안 함)
    dehydrateOptions: { shouldDehydrateQuery: defaultShouldDehydrateQuery },
  }}
>
  {children}
</PersistQueryClientProvider>
```

## 실험적 기능: Prefetch 없는 스트리밍

`@tanstack/react-query-next-experimental` 패키지를 사용하면 prefetch 없이도 스트리밍 SSR 가능:

```tsx
// app/providers.tsx
import { ReactQueryStreamedHydration } from '@tanstack/react-query-next-experimental'

export function Providers(props: { children: React.ReactNode }) {
  const queryClient = getQueryClient()

  return (
    <QueryClientProvider client={queryClient}>
      <ReactQueryStreamedHydration>
        {props.children}
      </ReactQueryStreamedHydration>
    </QueryClientProvider>
  )
}
```

**장점**: 
- 수동 prefetch 불필요
- 뛰어난 DX와 낮은 코드 복잡도
- 초기 페이지 로드 시 waterfall 제거

**단점**: 
- 페이지 네비게이션 시 request waterfall 발생
- getServerSideProps/getStaticProps보다 성능 저하 가능

```
1. |> JS for <Feed>
2.   |> getFeed()
3.     |> JS for <GraphFeedItem>
4.       |> getGraphDataById()
```

**사용 권장 케이스**:
- DX/개발 속도를 성능보다 우선시할 때
- 깊게 중첩된 쿼리가 없을 때
- useSuspenseQueries 등으로 병렬 fetching을 잘 관리할 때

## 핵심 요약

1. **Server Components는 prefetch 장소로만 사용**하고, 데이터 렌더링은 Client Components에서
2. **staleTime 설정** 필수 (SSR에서 즉시 재요청 방지)
3. **단일 QueryClient vs 여러 QueryClient**: 상황에 따라 선택
4. **Pending 쿼리 스트리밍**으로 초기 prefetch를 기다리지 않고 빠른 렌더링 가능
5. **데이터 소유권 분리**: Server/Client Component 간 데이터 동기화 문제 주의
6. **Waterfall 최소화**: parallel routes, useSuspenseQueries 활용
