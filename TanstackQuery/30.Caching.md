# Caching Examples (캐싱 예제)

> 이 가이드를 읽기 전에 Important Defaults를 먼저 읽어보세요

## 기본 예제

이 캐싱 예제는 다음의 스토리와 생명주기를 보여줍니다:
- 캐시 데이터가 있는/없는 Query 인스턴스
- 백그라운드 리페칭 (Background Refetching)
- 비활성 쿼리 (Inactive Queries)
- 가비지 컬렉션 (Garbage Collection)

기본 `gcTime`이 5분이고 기본 `staleTime`이 0이라고 가정합니다.

## 캐싱 생명주기

### 1단계: 첫 번째 쿼리 마운트
```javascript
useQuery({ queryKey: ['todos'], queryFn: fetchTodos })
```
- `['todos']` 키로 만들어진 다른 쿼리가 없으므로, 이 쿼리는 하드 로딩 상태를 표시하고 네트워크 요청을 시작
- 네트워크 요청이 완료되면, 반환된 데이터는 `['todos']` 키 아래에 캐시됨
- 데이터는 설정된 `staleTime` 이후 stale로 표시됨 (기본값 0, 즉시)

### 2단계: 두 번째 쿼리 인스턴스 마운트
```javascript
// 다른 곳에서 동일한 쿼리 마운트
useQuery({ queryKey: ['todos'], queryFn: fetchTodos })
```
- 캐시에 이미 `['todos']` 키의 데이터가 있으므로, 그 데이터가 캐시에서 **즉시 반환**됨
- 새 인스턴스는 query function을 사용하여 **새로운 네트워크 요청을 트리거**함
- 두 `fetchTodos` 함수가 동일하든 아니든, 같은 query key를 가지므로 **두 쿼리의 상태가 모두 업데이트**됨 (`isFetching`, `isPending` 등)
- 요청이 성공적으로 완료되면, `['todos']` 키 아래의 캐시 데이터가 새 데이터로 업데이트되고, **두 인스턴스 모두** 새 데이터로 업데이트됨

### 3단계: 모든 인스턴스 언마운트
- 두 쿼리 인스턴스가 모두 언마운트되고 더 이상 사용되지 않음
- 이 쿼리의 활성 인스턴스가 더 이상 없으므로, `gcTime`을 사용하여 **가비지 컬렉션 타임아웃이 설정**됨 (기본값 5분)

### 4단계: 타임아웃 전 재마운트
- 캐시 타임아웃(`gcTime`)이 완료되기 전에, 다른 인스턴스가 마운트됨
- 쿼리는 사용 가능한 **캐시된 데이터를 즉시 반환**하면서, 동시에 백그라운드에서 `fetchTodos` 함수를 실행함
- 백그라운드 요청이 성공적으로 완료되면, 캐시를 최신 데이터로 채움

### 5단계: 최종 인스턴스 언마운트
- 마지막 인스턴스가 언마운트됨
- 5분 동안 새로운 쿼리 인스턴스가 나타나지 않음
- `['todos']` 키 아래의 **캐시된 데이터가 삭제되고 가비지 컬렉션**됨

## 핵심 요점

### staleTime vs gcTime
- **`staleTime`**: 데이터가 얼마나 오래되면 "오래된(stale)" 것으로 간주할지 결정 (기본값: 0)
  - stale 데이터는 백그라운드에서 자동으로 리페칭됨
  
- **`gcTime`**: 비활성 쿼리가 메모리에서 제거되기까지 얼마나 기다릴지 결정 (기본값: 5분)
  - 활성 인스턴스가 없는 쿼리는 "비활성(inactive)" 상태

### 캐싱의 장점
1. **즉각적인 UI 응답**: 캐시된 데이터를 즉시 표시하므로 로딩 상태 감소
2. **백그라운드 업데이트**: 사용자가 오래된 데이터를 보는 동안 자동으로 최신 데이터를 가져옴
3. **네트워크 요청 최적화**: 동일한 데이터에 대한 중복 요청 방지
4. **메모리 관리**: gcTime을 통해 사용하지 않는 데이터를 자동으로 정리

### 같은 queryKey를 가진 여러 인스턴스
- 모든 인스턴스가 **동일한 캐시를 공유**함
- 하나의 인스턴스가 데이터를 업데이트하면 **모든 인스턴스가 업데이트**됨
- 상태(`isFetching`, `isPending` 등)도 **모든 인스턴스에 동기화**됨
