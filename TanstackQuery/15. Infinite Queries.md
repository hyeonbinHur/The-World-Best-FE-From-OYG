# Infinite Queries (무한 쿼리)

## 요점 정리

1. **`useInfiniteQuery`** 는 "더 보기" 또는 "무한 스크롤" 같은 점진적 데이터 로딩 UI를 구현하기 위한 `useQuery`의 확장판이다.
2. `data.pages` 배열에 가져온 페이지들이, `data.pageParams` 배열에 각 페이지를 가져올 때 사용한 파라미터가 저장된다.
3. **`getNextPageParam`** 으로 다음 페이지 존재 여부와 요청 정보를 결정하고, `fetchNextPage()`로 다음 페이지를 가져온다.
4. 동시 fetch 충돌을 방지하려면 `isFetching` 상태를 확인한 뒤 `fetchNextPage`를 호출해야 한다.
5. stale 상태에서 refetch 시 **첫 번째 페이지부터 순차적으로** 다시 가져온다.
6. **`maxPages`** 옵션으로 메모리에 보관하는 페이지 수를 제한하여 성능을 최적화할 수 있다.

---

## 개요

"더 보기(Load More)" 또는 "무한 스크롤(Infinite Scroll)" 패턴처럼, 기존 데이터 위에 추가 데이터를 점진적으로 로딩하는 리스트를 렌더링하는 것은 매우 흔한 UI 패턴이다. TanStack Query는 이러한 유형의 리스트를 위해 `useQuery`의 유용한 버전인 **`useInfiniteQuery`** 를 제공한다.

---

## `useQuery`와 달라지는 점

| 항목 | 설명 |
|---|---|
| `data.pages` | 가져온 페이지들의 배열 |
| `data.pageParams` | 각 페이지를 가져올 때 사용한 파라미터 배열 |
| `fetchNextPage` / `fetchPreviousPage` | 다음/이전 페이지를 가져오는 함수 (`fetchNextPage`는 필수) |
| `initialPageParam` | 초기 페이지 파라미터 (필수) |
| `getNextPageParam` / `getPreviousPageParam` | 추가 데이터 존재 여부 및 fetch 정보를 결정하는 옵션 |
| `hasNextPage` / `hasPreviousPage` | 다음/이전 페이지 존재 여부 (`null`·`undefined`가 아니면 `true`) |
| `isFetchingNextPage` / `isFetchingPreviousPage` | 백그라운드 새로고침과 추가 로딩 상태를 구분하는 boolean |

> **참고:** `initialData` 또는 `placeholderData`를 사용할 경우 `data.pages`와 `data.pageParams` 속성을 포함하는 동일한 객체 구조여야 한다.

---

## 예제 — "Load More" UI

커서 기반으로 프로젝트를 3개씩 반환하는 API가 있다고 가정:

```tsx
fetch('/api/projects?cursor=0')  // { data: [...], nextCursor: 3 }
fetch('/api/projects?cursor=3')  // { data: [...], nextCursor: 6 }
fetch('/api/projects?cursor=6')  // { data: [...], nextCursor: 9 }
fetch('/api/projects?cursor=9')  // { data: [...] }  ← 마지막 페이지
```

구현 흐름:
1. `useInfiniteQuery`가 기본적으로 첫 번째 데이터 그룹을 요청한다.
2. `getNextPageParam`에서 다음 쿼리 정보를 반환한다.
3. `fetchNextPage()` 함수를 호출하여 다음 페이지를 로딩한다.

```tsx
import { useInfiniteQuery } from '@tanstack/react-query'

function Projects() {
  const fetchProjects = async ({ pageParam }) => {
    const res = await fetch('/api/projects?cursor=' + pageParam)
    return res.json()
  }

  const {
    data,
    error,
    fetchNextPage,
    hasNextPage,
    isFetching,
    isFetchingNextPage,
    status,
  } = useInfiniteQuery({
    queryKey: ['projects'],
    queryFn: fetchProjects,
    initialPageParam: 0,
    getNextPageParam: (lastPage, pages) => lastPage.nextCursor,
  })

  return status === 'pending' ? (
    <p>Loading...</p>
  ) : status === 'error' ? (
    <p>Error: {error.message}</p>
  ) : (
    <>
      {data.pages.map((group, i) => (
        <React.Fragment key={i}>
          {group.data.map((project) => (
            <p key={project.id}>{project.name}</p>
          ))}
        </React.Fragment>
      ))}
      <div>
        <button
          onClick={() => fetchNextPage()}
          disabled={!hasNextPage || isFetching}
        >
          {isFetchingNextPage
            ? 'Loading more...'
            : hasNextPage
              ? 'Load More'
              : 'Nothing more to load'}
        </button>
      </div>
      <div>{isFetching && !isFetchingNextPage ? 'Fetching...' : null}</div>
    </>
  )
}
```

## 동시 fetch 주의사항

- 진행 중인 fetch가 있는 상태에서 `fetchNextPage`를 호출하면 백그라운드 데이터 새로고침을 덮어쓸 위험이 있다.
- **InfiniteQuery는 단일 캐시 항목을 공유**하므로, 동시에 두 번 fetch를 시도하면 데이터가 덮어써질 수 있다.
- 동시 fetch를 허용하려면 `{ cancelRefetch: false }` 옵션을 사용한다 (기본값: `true`).
- 권장: `isFetching` 상태를 확인한 뒤에 `fetchNextPage`를 호출한다.

```jsx
<List onEndReached={() => hasNextPage && !isFetching && fetchNextPage()} />
```

---

## Refetch 동작

무한 쿼리가 stale 상태가 되어 refetch가 필요하면, **첫 번째 그룹부터 순차적으로** 모든 그룹을 다시 가져온다. 이렇게 하면 기존 데이터가 변경되었더라도 오래된 커서를 사용하여 중복이나 누락이 발생하는 것을 방지할 수 있다. 쿼리 캐시에서 결과가 제거되면 페이지네이션이 초기 상태로 리셋된다.

---

## 양방향 무한 리스트 (Bi-directional)

`getPreviousPageParam`, `fetchPreviousPage`, `hasPreviousPage`, `isFetchingPreviousPage`를 사용하면 양방향 리스트를 구현할 수 있다.

```tsx
useInfiniteQuery({
  queryKey: ['projects'],
  queryFn: fetchProjects,
  initialPageParam: 0,
  getNextPageParam: (lastPage, pages) => lastPage.nextCursor,
  getPreviousPageParam: (firstPage, pages) => firstPage.prevCursor,
})
```

---

## 페이지 역순 표시

`select` 옵션을 사용하여 페이지 순서를 뒤집을 수 있다:

```tsx
useInfiniteQuery({
  queryKey: ['projects'],
  queryFn: fetchProjects,
  select: (data) => ({
    pages: [...data.pages].reverse(),
    pageParams: [...data.pageParams].reverse(),
  }),
})
```

## 무한 쿼리 수동 업데이트

### 첫 번째 페이지 제거

```tsx
queryClient.setQueryData(['projects'], (data) => ({
  pages: data.pages.slice(1),
  pageParams: data.pageParams.slice(1),
}))
```

### 개별 페이지에서 특정 항목 제거

```tsx
const newPagesArray =
  oldPagesArray?.pages.map((page) =>
    page.filter((val) => val.id !== updatedId),
  ) ?? []

queryClient.setQueryData(['projects'], (data) => ({
  pages: newPagesArray,
  pageParams: data.pageParams,
}))
```

### 첫 번째 페이지만 유지

```tsx
queryClient.setQueryData(['projects'], (data) => ({
  pages: data.pages.slice(0, 1),
  pageParams: data.pageParams.slice(0, 1),
}))
```

> **중요:** `pages`와 `pageParams`의 데이터 구조를 항상 동일하게 유지해야 한다!

## 페이지 수 제한 (Limited Infinite Query)

사용자가 매우 많은 페이지를 로드할 수 있는 경우 메모리와 네트워크 사용량을 줄이기 위해 **`maxPages`** 옵션을 사용할 수 있다. refetch 시에도 `maxPages`에 설정된 수만큼만 순차적으로 다시 가져온다.

```tsx
useInfiniteQuery({
  queryKey: ['projects'],
  queryFn: fetchProjects,
  initialPageParam: 0,
  getNextPageParam: (lastPage, pages) => lastPage.nextCursor,
  getPreviousPageParam: (firstPage, pages) => firstPage.prevCursor,
  maxPages: 3, // 최대 3페이지만 메모리에 보관
})
```

## API에 커서가 없는 경우

API가 커서를 반환하지 않으면 `pageParam` 자체를 커서로 활용할 수 있다. `getNextPageParam`과 `getPreviousPageParam`은 현재 페이지의 `pageParam`도 인자로 받으므로, 이를 사용하여 다음/이전 페이지 파라미터를 계산한다.

```tsx
return useInfiniteQuery({
  queryKey: ['projects'],
  queryFn: fetchProjects,
  initialPageParam: 0,
  getNextPageParam: (lastPage, allPages, lastPageParam) => {
    if (lastPage.length === 0) {
      return undefined // 더 이상 데이터 없음
    }
    return lastPageParam + 1
  },
  getPreviousPageParam: (firstPage, allPages, firstPageParam) => {
    if (firstPageParam <= 1) {
      return undefined
    }
    return firstPageParam - 1
  },
})
```
