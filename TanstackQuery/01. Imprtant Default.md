01 Important Default

- Stale Time = 데이터가 유효한 시간 -> stale time이 길수록 api 요청 빈도가 낮아진다.
  - Inifiity & static: 유효시간을 무제한으로 설정
    - Infinity: 필요할 때만 항상 수동으로 가져온다 (앱의 컨텐츠). -> 사용자 개인 설정 (theme), 확정된 과거 기록 (결제 내역) -> 뭐 하나 잘못나오면 해당 부분만 수동 변경해도 장애가 없는경우.
    - Static: 수동으로 조차 새로 요청하지 못하고, 항상 캐시에 있는데이터를 가져온다 (앱 설정). -> 환경변수, 배포 버전, 정적 참조 데이터 등 세션동안 변하지 않는 값들을 static하게 둔다. -> 해당 부분 하나 리로드한다고 해결되지 않고 앱 자체를 리부트시켜야 해결되는 경우
- 요청
  - refetch on mount, refetchOnWindowFocus, refetchOnReconnect 등을 이용해서, stale time이 만기된 데이터를 자동으로 요청 가능하다.
  - invalidateQuereis: stale time을 무시하고 수동으로 데이터를 가져온다.
- gcTime: stale time이 끝났지만, 당장 삭제하지 않고 잠깐 대기 시키는 시간
  - stale time이 끝나고 gcTime으로 넘어간 데이터는 [inactive]레이블이 붙은 상태로 캐시에 저장된다.
  - 로딩화면을 보여주지 않고 UX 향상을 위해 사용된다. (화면에 아예 로딩 스피너가 돌아가는 것과, 이전 데이터라도 보여주는건 UX상 큰 차이가 존재)
  - gcTime까지 끝나면 로딩 화면이 나온다.
- Retry: 쿼리 실패시 재시도 로직
  - 지수 백오프 (exponential backoff): 다음 retry까지 대기시간, 2^n 으로 시간이 늘어난다. Retry의 기본 아이디어는, 로직 오류가 아닌 서버 부하로 인해 실패하는걸 대기하기 때문에, 일정 시간을 두고 Retry를 시도한다. 꼭 2^n 일 필요는 없지만 IT업계 네트워크 통신 규약이 역사적으로 2진 짓수 백오프를 표준처럼 사용해왔기 때문에 관습적으로 2^n 을 따른다. -> 지수 백오프 시간 설정도 가능하지만 기본 스펙상 최대 30초를 넘기지 않는다 (커스텀이 가능하긴합니다).
  - 3번 retry가 기본이고, retry: true를 이용해서 무한정 재시도가 가능하긴 하다.
- Structural Sharing: 구조적 공유 (서버에서 새로운 값을 가져와서 기존 데이터를 덮어 씌울때)
  - refetching해서 새로운 데이터를 가져와서 기존 데이터와 비교한후, 변경된 값만 새로 정의한다.
  - 기본적으로 데이터는 값 참조가 아닌 referency 참조이기 때문에, reference를 비교하는 useMemo, useCallback를 통한 최적화에도 문제없이 사용 가능합니다.
    - Json 데이터 구조적 공유는 원할하게 가능하지만, Map, Set, Date, Class같은 인스턴스는 공유시 원본의 성질을 잃을 수 있기 때문에 안전하게 비교하지 못합니다.
  - ex) Todo List Query를 날려서 Todd[] 배열 100개 값을 가져온다
    ![[Pasted image 20260412150314.png]]
  - TodoList.tsx > Todo.tsx로 100개를 렌더링한다
  - Todo 중 하나가 바뀌었고, TodoListQuery를 다시 날려서 100개를 받아온다

  - 이를 방지하기 비교해서 바뀐 데이터만 새로운 reference로 넣어주고, 99개는 기존 referece를 유지한다.
