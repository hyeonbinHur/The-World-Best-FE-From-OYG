폴링(Polling)은 “일정 시간마다 서버에 데이터를 다시 요청하는 것”입니다.

refetchInterval 옵션으로 설정이 가능하며 코드는 아래와 같습니다.

```tsx
useQuery({
  queryKey: ["prices"],
  queryFn: fetchPrices,
  refetchInterval: 5_000, // 5초마다 다시 가져옴
});
```

폴링은 `staleTime`이랑 **별개로 동작**합니다. `staleTime`은 “데이터가 신선한지 오래된지”를 판단하는 건데, 폴링은 데이터가 신선하든 말든 무조건 타이머대로 실행됩니다. 쉽게 말해 `staleTime: 60_000`으로 1분 동안 신선하다고 설정해도, `refetchInterval: 5_000`이면 5초마다 어차피 다시 가져온다는 뜻.

### 쿼리 상태에 따른 간격 조절

숫자 대신 함수 넘기기

```tsx
useQuery({
  queryKey: ["job", jobId],
  queryFn: () => fetchJobStatus(jobId),
  refetchInterval: (query) => {
    // 작업이 완료되면 폴링 중단
    if (query.state.data?.status === "complete") return false;
    return 2_000; // 아직 진행 중이면 2초마다
  },
});
```

query는 queryFn의 결과를 가지고 있는 tanstackQuery 내부 함수입니다.

```tsx
query = {
  state: {
    data: { status: 'complete', ... },  // ← queryFn이 리턴한 데이터
    error: null,
    status: 'success',                  // 쿼리 자체의 상태 (pending/success/error)
    fetchStatus: 'idle',
    ...
  },
  queryKey: ['job', jobId],
  ...
}
```

## 백그라운드 폴링

기본적으로 **브라우저 탭이 비활성화되면 폴링이 멈춥니다**. 근데 대시보드처럼 다른 탭에 있어도 데이터가 계속 최신이어야 하는 경우엔 `refetchIntervalInBackground: true`를 추가해 다른 탭이더라도, focus 아웃되더라도 계속 폴링시킬수 있습니다.

```tsx
useQuery({
  queryKey: ["portfolio"],
  queryFn: fetchPortfolio,
  refetchInterval: 30_000, // 30초마다
  refetchIntervalInBackground: true, // 탭 안 보고 있어도 계속 폴링
});
```

## 폴링 일시정지

함수 안에서 컴포넌트의 state를 참조해서 폴링을 켜고 끌 수 있습니다.

```tsx
const [isPaused, setIsPaused] = useState(false);

useQuery({
  queryKey: ["prices", tokenAddress],
  queryFn: () => fetchPrice(tokenAddress),
  refetchInterval: () => {
    if (!tokenAddress || isPaused) return false; // 주소 없거나 일시정지면 폴링 중단
    return 15_000; // 15초마다
  },
});
```

## 오프라인 환경에서 폴링

netowork mode가 online, offlinefirst에서는 폴링이 동작하지 않으니까 always로 해주고 polling 합니다.

```tsx
useQuery({
  queryKey: ["chainStatus"],
  queryFn: fetchChainStatus,
  refetchInterval: 10_000,
  networkMode: "always", // 온라인/오프라인 체크 안 함
});
```

## 중복 제거(Deduplication)에 대한 참고사항

같은 `queryKey: ['prices']`를 사용하는 컴포넌트가 2개 마운트되어 있습니다:

```tsx
<PriceCard />  // useQuery + refetchInterval: 5000
<PriceCard />  // useQuery + refetchInterval: 5000
```

타이머는 각자 돌아갑니다

TanStack Query에서 `useQuery`를 호출하면 내부적으로 `QueryObserver`라는 것이 하나 생깁니다. 컴포넌트가 2개이니까 옵저버도 2개, **타이머도 2개**입니다. 이 타이머들은 서로의 존재를 모르고 독립적으로 돌아갑니다.

하지만 네트워크 요청은 queryKey 단위로 관리됩니다

컴포넌트A 타이머 발동 → "['prices'] fetch 해야지"
→ QueryCache 확인: 이 키로 날아가고 있는 요청이 있는가?
→ 없음 → fetch 실행 ✅

컴포넌트B 타이머 발동 → "['prices'] fetch 해야지"
(거의 동시) → QueryCache 확인: 이 키로 날아가고 있는 요청이 있는가?
→ 있음 → 새 요청을 보내지 않고 기존 요청에 편승 🚫
응답이 도착하면

A가 트리거한 요청의 응답이 도착하면, **같은 queryKey를 구독하고 있는 모든 컴포넌트**(A, B 둘 다)가 새 데이터를 받아서 리렌더링됩니다.
