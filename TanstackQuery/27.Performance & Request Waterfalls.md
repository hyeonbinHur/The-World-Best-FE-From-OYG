# Performance & Request Waterfalls (성능 및 요청 워터폴)

## 개요
애플리케이션 성능은 광범위하고 복잡한 영역입니다. React Query가 API를 더 빠르게 만들 수는 없지만, React Query를 사용하는 방식에서 최상의 성능을 보장하기 위해 유의해야 할 사항들이 있습니다.

React Query나 컴포넌트 내부에서 데이터를 가져올 수 있게 해주는 데이터 페칭 라이브러리를 사용할 때 가장 큰 성능 문제는 **요청 워터폴(Request Waterfalls)**입니다.

## Request Waterfall이란?

### 정의
요청 워터폴은 어떤 리소스(코드, CSS, 이미지, 데이터)에 대한 요청이 다른 리소스에 대한 요청이 완료된 후에야 시작되는 현상입니다.

### 예시
웹 페이지를 생각해보면, CSS, JS 등을 로드하기 전에 브라우저는 먼저 마크업을 로드해야 합니다:

```
1. |-> Markup
2.   |-> CSS
2.   |-> JS
2.   |-> Image
```

JS 파일 내에서 CSS를 가져오면 이중 워터폴이 발생합니다:

```
1. |-> Markup
2.   |-> JS
3.     |-> CSS
```

CSS가 배경 이미지를 사용하면 삼중 워터폴이 발생합니다:

```
1. |-> Markup
2.   |-> JS
3.     |-> CSS
4.       |-> Image
```

### 성능 영향
- 각 워터폴은 서버에 대한 최소 한 번의 왕복을 나타냅니다
- 250ms 지연시간(3G 네트워크나 나쁜 네트워크 환경에서 흔함)에서:
  - 4번의 왕복 = 4 × 250ms = 1000ms
  - 2번의 왕복 = 2 × 250ms = 500ms
- 워터폴을 평평하게 만들면 로딩 시간을 절반으로 줄일 수 있습니다!

### 워터폴 감지 방법
브라우저 개발자 도구의 "Network" 탭을 사용하는 것이 가장 좋은 방법입니다.

## React Query와 Request Waterfalls

### 기본 구조
서버 렌더링이 없는 경우, React Query로 쿼리를 시작하기 전에 JS를 로드해야 하므로:

```
1. |-> Markup
2.   |-> JS
3.     |-> Query
```

## 워터폴 패턴과 해결 방법

### 1. Single Component Waterfalls / Serial Queries (단일 컴포넌트 워터폴)

#### 문제 상황
하나의 컴포넌트가 먼저 하나의 쿼리를 가져온 다음 다른 쿼리를 가져올 때 발생합니다.

**Dependent Query 예시:**
```tsx
// 사용자 가져오기
const { data: user } = useQuery({
  queryKey: ['user', email],
  queryFn: getUserByEmail,
})

const userId = user?.id

// 사용자의 프로젝트 가져오기
const { data: projects } = useQuery({
  queryKey: ['projects', userId],
  queryFn: getProjectsByUser,
  enabled: !!userId, // userId가 있을 때만 실행
})
```

#### 해결 방법
1. **API 재구조화**: 두 쿼리를 하나로 합치기
   - `getProjectsByUserEmail` 같은 새로운 쿼리 도입
2. **서버로 워터폴 이동**: Server Components 활용 (서버에서는 지연시간이 낮음)

#### Suspense를 사용한 Serial Queries

**문제:**
```tsx
function App() {
  // 다음 쿼리들은 순차적으로 실행되어 서버에 대한 별도의 왕복을 발생시킵니다:
  const usersQuery = useSuspenseQuery({ queryKey: ['users'], queryFn: fetchUsers })
  const teamsQuery = useSuspenseQuery({ queryKey: ['teams'], queryFn: fetchTeams })
  const projectsQuery = useSuspenseQuery({ queryKey: ['projects'], queryFn: fetchProjects })
}
```

**해결:**
```tsx
const [usersQuery, teamsQuery, projectsQuery] = useSuspenseQueries({
  queries: [
    { queryKey: ['users'], queryFn: fetchUsers },
    { queryKey: ['teams'], queryFn: fetchTeams },
    { queryKey: ['projects'], queryFn: fetchProjects },
  ],
})
```

### 2. Nested Component Waterfalls (중첩 컴포넌트 워터폴)

#### 문제 상황
부모와 자식 컴포넌트 모두 쿼리를 포함하고, 부모가 쿼리를 완료할 때까지 자식을 렌더링하지 않는 경우입니다.

**독립적인 경우 - 문제:**
```tsx
function Article({ id }) {
  const { data: articleData, isPending } = useQuery({
    queryKey: ['article', id],
    queryFn: getArticleById,
  })

  if (isPending) {
    return 'Loading article...'
  }

  return (
    <>
      <ArticleHeader articleData={articleData} />
      <ArticleBody articleData={articleData} />
      <Comments id={id} /> {/* id는 이미 있지만 나중에 fetch */}
    </>
  )
}

function Comments({ id }) {
  const { data, isPending } = useQuery({
    queryKey: ['article-comments', id],
    queryFn: getArticleCommentsById,
  })
  // ...
}
```

**해결 - 쿼리 호이스팅:**
```tsx
function Article({ id }) {
  const { data: articleData, isPending: articlePending } = useQuery({
    queryKey: ['article', id],
    queryFn: getArticleById,
  })

  const { data: commentsData, isPending: commentsPending } = useQuery({
    queryKey: ['article-comments', id],
    queryFn: getArticleCommentsById,
  })

  if (articlePending) {
    return 'Loading article...'
  }

  return (
    <>
      <ArticleHeader articleData={articleData} />
      <ArticleBody articleData={articleData} />
      {commentsPending ? (
        'Loading comments...'
      ) : (
        <Comments commentsData={commentsData} />
      )}
    </>
  )
}
```

#### 의존적인 Nested Component Waterfall

**문제:**
```tsx
function Feed() {
  const { data, isPending } = useQuery({
    queryKey: ['feed'],
    queryFn: getFeed,
  })

  if (isPending) {
    return 'Loading feed...'
  }

  return (
    <>
      {data.map((feedItem) => {
        if (feedItem.type === 'GRAPH') {
          return <GraphFeedItem key={feedItem.id} feedItem={feedItem} />
        }
        return <StandardFeedItem key={feedItem.id} feedItem={feedItem} />
      })}
    </>
  )
}

function GraphFeedItem({ feedItem }) {
  const { data, isPending } = useQuery({
    queryKey: ['graph', feedItem.id],
    queryFn: getGraphDataById,
  })
  // ...
}
```

워터폴:
```
1. |> getFeed()
2.   |> getGraphDataById()
```

**해결 방법:**
1. API 재구조화: `getFeed` 쿼리에 그래프 데이터 포함
2. Server Components 활용: 서버로 워터폴 이동 (고급 아키텍처 변경)

### 3. Code Splitting (코드 분할)

#### 문제 상황
코드 분할은 성능에 중요하지만, 요청 워터폴을 도입할 수 있습니다.

**문제:**
```tsx
// GraphFeedItem 컴포넌트를 지연 로드
const GraphFeedItem = React.lazy(() => import('./GraphFeedItem'))

function Feed() {
  const { data, isPending } = useQuery({
    queryKey: ['feed'],
    queryFn: getFeed,
  })

  if (isPending) {
    return 'Loading feed...'
  }

  return (
    <>
      {data.map((feedItem) => {
        if (feedItem.type === 'GRAPH') {
          return <GraphFeedItem key={feedItem.id} feedItem={feedItem} />
        }
        return <StandardFeedItem key={feedItem.id} feedItem={feedItem} />
      })}
    </>
  )
}
```

이중 워터폴:
```
1. |> getFeed()
2.   |> JS for <GraphFeedItem>
3.     |> getGraphDataById()
```

첫 페이지 로드 시 실제로는 5번의 왕복:
```
1. |> Markup
2.   |> JS for <Feed>
3.     |> getFeed()
4.       |> JS for <GraphFeedItem>
5.         |> getGraphDataById()
```

#### 해결 방법
`getGraphDataById` 쿼리를 `<Feed>` 컴포넌트로 호이스팅하거나 조건부 prefetch 추가:

```
1. |> getFeed()
2.   |> getGraphDataById()
2.   |> JS for <GraphFeedItem>
```

#### 트레이드오프
- **옵션 1**: 모든 데이터 페칭 코드를 메인 번들에 포함 (거의 사용하지 않아도)
- **옵션 2**: 데이터 페칭 코드를 코드 분할 번들에 포함하지만 요청 워터폴 발생

이 트레이드오프는 Server Components의 동기 중 하나였습니다. Server Components를 사용하면 두 가지 모두 피할 수 있습니다.

## 요약 및 핵심 사항

### Request Waterfalls는 매우 흔하고 복잡한 성능 문제입니다

**워터폴이 실수로 도입되는 경우:**
- 부모가 이미 쿼리를 가지고 있다는 것을 모르고 자식에 쿼리 추가
- 자식이 이미 쿼리를 가지고 있다는 것을 모르고 부모에 쿼리 추가
- 쿼리가 있는 후손을 가진 컴포넌트를 쿼리가 있는 조상이 있는 새 부모로 이동

### 모범 사례
1. **정기적으로 Network 탭 검사**: 워터폴을 찾기 위해 정기적으로 애플리케이션 검사
2. **모든 워터폴을 평평하게 만들 필요는 없음**: 좋은 성능을 위해 모든 것을 제거할 필요는 없지만, 영향이 큰 것들을 주시하세요
3. **다음 단계**: Prefetching & Router Integration 가이드에서 더 많은 워터폴 평평화 방법 학습

## 핵심 포인트
- ✅ Request Waterfalls는 리소스 요청이 다른 요청 완료 후 시작되는 현상
- ✅ 지연시간에 따라 성능에 큰 영향을 미침 (각 워터폴 = 서버 왕복)
- ✅ 주요 원인: Serial Queries, Nested Components, Code Splitting
- ✅ 해결: API 재구조화, 쿼리 호이스팅, useSuspenseQueries 사용, Prefetching
- ✅ Server Components로 워터폴을 서버로 이동 가능 (지연시간 감소)
- ✅ Network 탭으로 정기적으로 모니터링하고 영향이 큰 워터폴에 집중
