# Server Rendering & Hydration (서버 렌더링 & 하이드레이션)

## 핵심 개념

### 서버 렌더링이란?
서버에서 초기 HTML을 생성하여 사용자가 페이지 로드 시 즉시 콘텐츠를 볼 수 있게 하는 것

**클라이언트 렌더링:**
```
1. |-> Markup (콘텐츠 없음)
2.   |-> JS
3.     |-> Query
```

**서버 렌더링:**
```
1. |-> Markup (콘텐츠 + 초기 데이터 포함)
2.   |-> JS
```

### 핵심 단계
1. **Prefetch**: 서버에서 데이터 미리 가져오기
2. **Dehydrate**: 데이터를 직렬화 가능한 형태로 변환
3. **Hydrate**: 클라이언트에서 React Query 캐시로 복원

## 초기 설정

### QueryClient 생성
**중요**: 각 요청마다 새로운 queryClient를 생성해야 함 (파일 루트 레벨에서 생성하면 안 됨!)

```tsx
// ❌ 절대 이렇게 하지 마세요
// const queryClient = new QueryClient()

// ✅ 올바른 방법 - 컴포넌트 내부에서 생성
export default function MyApp({ Component, pageProps }) {
  const [queryClient] = React.useState(
    () => new QueryClient({
      defaultOptions: {
        queries: {
          // SSR에서는 클라이언트에서 즉시 refetch를 방지하기 위해
          // staleTime을 0보다 크게 설정하는 것이 좋음
          staleTime: 60 * 1000,
        },
      },
    }),
  )

  return (
    <QueryClientProvider client={queryClient}>
      <Component {...pageProps} />
    </QueryClientProvider>
  )
}
```

**이유**: 파일 루트에서 생성하면 모든 사용자 간에 캐시가 공유되어 성능 문제와 보안 문제 발생

## 방법 1: initialData 사용 (빠른 시작)

### 장점
- 설정이 간단함
- 빠르게 시작 가능

### 단점
- 깊은 컴포넌트 트리에서는 props drilling 필요
- 여러 곳에서 같은 쿼리를 사용할 때 관리가 어려움
- 데이터 fetch 시점을 알 수 없음
- 캐시에 이미 데이터가 있으면 새 데이터로 덮어쓰지 않음

```tsx
// Next.js 예시
export async function getServerSideProps() {
  const posts = await getPosts()
  return { props: { posts } }
}

function Posts(props) {
  const { data } = useQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
    initialData: props.posts,
  })
}
```

## 방법 2: Hydration API 사용 (권장)

### 설정 단계

1. **Loader 함수에서 QueryClient 생성**
```tsx
const queryClient = new QueryClient(options)
```

2. **쿼리 Prefetch**
```tsx
await queryClient.prefetchQuery({
  queryKey: ['posts'],
  queryFn: getPosts,
})
```

3. **Dehydrate 후 반환**
```tsx
return {
  props: {
    dehydratedState: dehydrate(queryClient),
  },
}
```

4. **HydrationBoundary로 감싸기**
```tsx
<HydrationBoundary state={dehydratedState}>
  <Component />
</HydrationBoundary>
```

### Next.js Pages Router 전체 예시

```tsx
// pages/posts.tsx
export async function getStaticProps() {
  const queryClient = new QueryClient()

  await queryClient.prefetchQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
  })

  return {
    props: {
      dehydratedState: dehydrate(queryClient),
    },
  }
}

function Posts() {
  const { data } = useQuery({ 
    queryKey: ['posts'], 
    queryFn: getPosts 
  })
  
  // 이 쿼리는 서버에서 prefetch하지 않음
  // 클라이언트에서만 fetch됨
  const { data: commentsData } = useQuery({
    queryKey: ['posts-comments'],
    queryFn: getComments,
  })
}

export default function PostsRoute({ dehydratedState }) {
  return (
    <HydrationBoundary state={dehydratedState}>
      <Posts />
    </HydrationBoundary>
  )
}
```

### Remix 예시

```tsx
// app/routes/posts.tsx
export async function loader() {
  const queryClient = new QueryClient()

  await queryClient.prefetchQuery({
    queryKey: ['posts'],
    queryFn: getPosts,
  })

  return json({ dehydratedState: dehydrate(queryClient) })
}

export default function PostsRoute() {
  const { dehydratedState } = useLoaderData<typeof loader>()
  return (
    <HydrationBoundary state={dehydratedState}>
      <Posts />
    </HydrationBoundary>
  )
}
```

## Dependent Queries Prefetch

```tsx
export async function getServerSideProps() {
  const queryClient = new QueryClient()

  // 첫 번째 쿼리
  const user = await queryClient.fetchQuery({
    queryKey: ['user', email],
    queryFn: getUserByEmail,
  })

  // 조건부 두 번째 쿼리
  if (user?.userId) {
    await queryClient.prefetchQuery({
      queryKey: ['projects', userId],
      queryFn: getProjectsByUser,
    })
  }

  return { props: { dehydratedState: dehydrate(queryClient) } }
}
```

## 에러 처리

### 기본 동작
- `prefetchQuery`: 에러를 throw하지 않음
- `dehydrate`: 성공한 쿼리만 포함, 실패한 쿼리는 제외
- 실패한 쿼리는 클라이언트에서 재시도됨

### 중요한 콘텐츠에 대한 에러 처리
```tsx
try {
  result = await queryClient.fetchQuery(...)
} catch (error) {
  // 404나 500 응답 처리
}
```

### 실패한 쿼리도 포함하기
```tsx
dehydrate(queryClient, {
  shouldDehydrateQuery: (query) => {
    // 실패한 쿼리도 포함
    return true
  },
})
```

## Boilerplate 제거 (선택사항)

매 라우트마다 HydrationBoundary를 반복하기 싫다면:

```tsx
// _app.tsx
export default function MyApp({ Component, pageProps }) {
  const [queryClient] = React.useState(() => new QueryClient())

  return (
    <QueryClientProvider client={queryClient}>
      <HydrationBoundary state={pageProps.dehydratedState}>
        <Component {...pageProps} />
      </HydrationBoundary>
    </QueryClientProvider>
  )
}
```

## Serialization (직렬화)

### 기본 제약사항
프레임워크는 기본적으로 안전하게 직렬화 가능한 것만 지원:
- ❌ undefined, Error, Date, Map, Set, BigInt, Infinity, NaN, -0, 정규표현식 등 불가

### 해결 방법
- `superjson` 같은 라이브러리 사용
- 커스텀 SSR 설정: `Serialize JavaScript`나 `devalue` 사용 (XSS 공격 방지)

## Request Waterfalls

### 클라이언트 렌더링
```
1. |> Markup (콘텐츠 없음)
2.   |> JS for <Feed>
3.     |> getFeed()
4.       |> JS for <GraphFeedItem>
5.         |> getGraphDataById()
```

### 서버 렌더링 (초기 로드)
```
1. |> Markup (콘텐츠 + 초기 데이터)
2.   |> JS for <Feed>
2.   |> JS for <GraphFeedItem>
```

### SPA 네비게이션
```
1. |> JS for <Feed>
1. |> getFeed() + getGraphDataById()
2.   |> JS for <GraphFeedItem>
```

**주의**: 서버 렌더링은 초기 페이지 로드에만 적용됨. 이후 네비게이션은 클라이언트 렌더링

## Tips & Caveats

### 1. Staleness 측정
- 서버에서 데이터를 가져온 시점부터 측정
- `staleTime` 기본값이 0이므로 페이지 로드 시 백그라운드에서 refetch됨
- SSR에서는 더 높은 `staleTime` 설정 권장

### 2. 서버 메모리 소비
- 요청마다 QueryClient 생성 시 메모리 소비 증가
- 서버에서 `gcTime` 기본값은 `Infinity`
- 해결: `queryClient.clear()` 호출하거나 더 작은 `gcTime` 설정
- **주의**: `gcTime`을 0으로 설정하면 hydration 에러 발생 가능 (최소 `2 * 1000` 권장)

### 3. Next.js Rewrites 주의사항
- Automatic Static Optimization이나 getStaticProps와 함께 사용 시
- React Query의 두 번째 hydration 발생
- 참조 동등성(referential equality) 손실 가능

## Suspense 사용

- `useQuery` 대신 `useSuspenseQuery` 사용 가능
- 모든 쿼리를 반드시 prefetch해야 함
- Prefetch 누락 시:
  - 서버에서 fetch되지만 클라이언트에 hydrate되지 않음
  - 클라이언트에서 다시 fetch
  - Markup hydration mismatch 발생

## 요약

1. ✅ 각 요청마다 새로운 QueryClient 생성
2. ✅ SSR에서는 `staleTime > 0` 설정 권장
3. ✅ Hydration API 사용 (dehydrate/hydrate)
4. ✅ 중요한 데이터는 `fetchQuery`로 에러 처리
5. ✅ Dependent queries도 prefetch 가능
6. ⚠️ 직렬화 제약사항 고려
7. ⚠️ 서버 메모리 관리 주의
8. ⚠️ 초기 로드 후 네비게이션은 클라이언트 렌더링
