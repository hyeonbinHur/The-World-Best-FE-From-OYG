# 21. Mutation 응답을 이용한 쿼리 업데이트 (Updates from Mutation Responses)

## 핵심 개념

서버에서 객체를 수정하는 Mutation을 수행하면, 응답으로 수정된 새 객체가 반환되는 경우가 많다. 이때 해당 데이터를 다시 fetch할 필요 없이, **Mutation 응답 데이터를 직접 캐시에 반영**하여 불필요한 네트워크 요청을 줄일 수 있다.

이를 위해 `QueryClient`의 `setQueryData` 메서드를 사용한다.

---

## 기본 사용법

Mutation의 `onSuccess` 콜백에서 `setQueryData`를 호출하여, 성공 시 반환된 데이터를 해당 쿼리 캐시에 즉시 반영한다.

```tsx
const queryClient = useQueryClient()

const mutation = useMutation({
  mutationFn: editTodo,
  onSuccess: (data) => {
    queryClient.setQueryData(['todo', { id: 5 }], data)
  },
})

mutation.mutate({
  id: 5,
  name: 'Do the laundry',
})

// 아래 쿼리는 Mutation 성공 응답 데이터로 자동 업데이트된다
const { status, data, error } = useQuery({
  queryKey: ['todo', { id: 5 }],
  queryFn: fetchTodoById,
})
```

---

## 재사용 가능한 커스텀 훅으로 만들기

`onSuccess` 로직을 재사용하려면 커스텀 훅으로 추출할 수 있다. `onSuccess`의 두 번째 인자 `variables`를 활용하면 동적으로 queryKey를 구성할 수 있다.

```tsx
const useMutateTodo = () => {
  const queryClient = useQueryClient()

  return useMutation({
    mutationFn: editTodo,
    // 두 번째 인자 variables는 mutate 함수가 받은 값이다
    onSuccess: (data, variables) => {
      queryClient.setQueryData(['todo', { id: variables.id }], data)
    },
  })
}
```

---

## 불변성 (Immutability)

`setQueryData`로 캐시를 업데이트할 때는 반드시 **불변(immutable) 방식**으로 수행해야 한다. 캐시에서 가져온 데이터를 직접 수정(mutate)하면 처음에는 동작하는 것처럼 보이지만, 미묘한 버그가 발생할 수 있다.

```tsx
// 잘못된 방법 - 직접 수정
queryClient.setQueryData(['posts', { id }], (oldData) => {
  if (oldData) {
    oldData.title = 'my new post title' // 직접 수정하면 안 된다
  }
  return oldData
})

// 올바른 방법 - 스프레드 연산자로 새 객체 생성
queryClient.setQueryData(
  ['posts', { id }],
  (oldData) =>
    oldData
      ? {
          ...oldData,
          title: 'my new post title',
        }
      : oldData,
)
```

## 요점 정리

| 항목 | 설명 |
|------|------|
| 목적 | Mutation 응답 데이터를 활용하여 불필요한 refetch 없이 캐시를 즉시 업데이트 |
| 핵심 메서드 | `queryClient.setQueryData(queryKey, data)` |
| 사용 위치 | `useMutation`의 `onSuccess` 콜백 내부 |
| 재사용 | 커스텀 훅으로 추출하여 `variables`를 활용한 동적 queryKey 구성 가능 |
| 불변성 규칙 | 캐시 데이터를 직접 수정하지 말고, 반드시 새 객체를 생성하여 반환해야 한다 |
| vs Invalidation | Invalidation은 캐시를 무효화하고 다시 fetch하지만, 이 방식은 응답 데이터를 직접 캐시에 반영하여 네트워크 요청을 절약한다 |
