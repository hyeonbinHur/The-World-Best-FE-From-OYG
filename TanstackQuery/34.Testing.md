# Testing

## 📌 핵심 요약

React Query는 훅 기반으로 작동하므로 테스트 시 특별한 설정이 필요하다. 각 테스트는 독립적인 `QueryClient`와 `QueryClientProvider`로 감싸야 하며, retry를 끄고 적절한 waitFor를 사용해야 한다.

---

## 1. 테스트 환경 설정

### React 버전별 라이브러리

**React 17 이하**
```bash
npm install @testing-library/react-hooks react-test-renderer --save-dev
```
- `@testing-library/react-hooks` 사용
- `react-test-renderer`는 peer dependency로 필요

**React 18 이상**
- `@testing-library/react`에서 `renderHook`을 직접 제공
- `@testing-library/react-hooks`는 더 이상 필요 없음

---

## 2. 첫 번째 테스트 작성

### Custom Hook 예시
```tsx
export function useCustomHook() {
  return useQuery({ queryKey: ['customHook'], queryFn: () => 'Hello' })
}
```

### 테스트 코드
```tsx
import { renderHook, waitFor } from '@testing-library/react'

const queryClient = new QueryClient()
const wrapper = ({ children }) => (
  <QueryClientProvider client={queryClient}>{children}</QueryClientProvider>
)

const { result } = renderHook(() => useCustomHook(), { wrapper })

await waitFor(() => expect(result.current.isSuccess).toBe(true))

expect(result.current.data).toEqual('Hello')
```

### 핵심 포인트
- **wrapper 제공**: 각 테스트마다 독립적인 `QueryClient`와 `QueryClientProvider` 생성
- **테스트 격리**: wrapper를 재사용할 경우 각 테스트 전에 `QueryClient`를 초기화해야 하며, 병렬 테스트는 피해야 함

---

## 3. Retry 끄기

React Query는 기본적으로 **3번의 재시도**와 **지수 백오프**를 사용한다. 에러 케이스를 테스트할 때 타임아웃이 발생할 수 있으므로 retry를 꺼야 한다.

```tsx
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      // ✅ retry 끄기
      retry: false,
    },
  },
})
const wrapper = ({ children }) => (
  <QueryClientProvider client={queryClient}>{children}</QueryClientProvider>
)
```

### 주의사항
- 이 설정은 **기본값**으로만 작동
- 개별 `useQuery`에서 명시적으로 `retry: 5`를 설정한 경우 해당 값이 우선 적용됨

---

## 4. Jest에서 gcTime 설정

Jest 사용 시 `gcTime`을 `Infinity`로 설정하면 "Jest did not exit one second after the test run completed" 에러를 방지할 수 있다.

- 서버에서는 기본값이 `Infinity`
- 명시적으로 `gcTime`을 설정한 경우에만 필요

---

## 5. 네트워크 호출 테스트

React Query의 주요 용도는 네트워크 요청을 캐싱하는 것이므로, 올바른 네트워크 요청이 이루어지는지 테스트해야 한다.

### nock을 사용한 예시

```tsx
function useFetchData() {
  return useQuery({
    queryKey: ['fetchData'],
    queryFn: () => request('/api/data'),
  })
}
```

### 테스트 코드
```tsx
const queryClient = new QueryClient()
const wrapper = ({ children }) => (
  <QueryClientProvider client={queryClient}>{children}</QueryClientProvider>
)

const expectation = nock('http://example.com').get('/api/data').reply(200, {
  answer: 42,
})

const { result } = renderHook(() => useFetchData(), { wrapper })

await waitFor(() => expect(result.current.isSuccess).toBe(true))

expect(result.current.data).toEqual({ answer: 42 })
```

### 핵심 포인트
- `waitFor`를 사용해 쿼리 상태가 성공으로 변경될 때까지 대기
- 요청이 완료되었을 때 올바른 데이터가 있는지 확인
- React 18에서는 `waitFor`의 동작 방식이 변경됨

## 6. Load More / Infinite Scroll 테스트

### API 응답 Mock 생성
```tsx
function generateMockedResponse(page) {
  return {
    page: page,
    items: [...]
  }
}
```

### nock 설정
```tsx
const expectation = nock('http://example.com')
  .persist()  // 여러 번 호출되므로 persist 필요
  .query(true)
  .get('/api/data')
  .reply(200, (uri) => {
    const url = new URL(`http://example.com${uri}`)
    const { page } = Object.fromEntries(url.searchParams)
    return generateMockedResponse(page)
  })
```

### 테스트 코드
```tsx
const { result } = renderHook(() => useInfiniteQueryCustomHook(), {
  wrapper,
})

await waitFor(() => expect(result.current.isSuccess).toBe(true))

expect(result.current.data.pages).toStrictEqual(generateMockedResponse(1))

result.current.fetchNextPage()

await waitFor(() =>
  expect(result.current.data.pages).toStrictEqual([
    ...generateMockedResponse(1),
    ...generateMockedResponse(2),
  ]),
)

expectation.done()
```

### 핵심 포인트
- `.persist()`를 사용해 동일 엔드포인트를 여러 번 호출할 수 있게 함
- `uri`를 통해 페이지별로 다른 응답을 반환
- `fetchNextPage()` 호출 후 `waitFor`로 데이터 변경 대기
