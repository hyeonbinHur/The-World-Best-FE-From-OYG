# notifyManager

## 개요
notifyManager는 TanStack Query에서 콜백의 스케줄링과 배칭(일괄 처리)을 담당하는 매니저입니다.

---

## 핵심 요약

### notifyManager의 역할
- **콜백 스케줄링**: 언제 콜백이 실행될지 관리
- **배칭 처리**: 여러 업데이트를 묶어서 한 번에 처리 → 성능 최적화
- **프레임워크 통합**: React, Solid 등 다양한 프레임워크와 연동 가능

### 주요 메서드 6가지
1. `batch` - 업데이트를 묶어서 처리
2. `batchCalls` - 함수 호출을 배치로 예약
3. `schedule` - 다음 배치에서 실행되도록 예약
4. `setNotifyFunction` - 알림 함수 커스터마이징
5. `setBatchNotifyFunction` - 배치 업데이트 함수 설정
6. `setScheduler` - 배치 실행 타이밍 설정

---

## 메서드 상세 설명

### 1. notifyManager.batch
**용도**: 콜백 내부에서 예약된 모든 업데이트를 일괄 처리

```typescript
function batch<T>(callback: () => T): T
```

**사용 사례**:
- TanStack Query 내부적으로 queryClient 업데이트 최적화에 사용
- 여러 상태 변경을 하나의 렌더링으로 묶을 때

**쉽게 풀어서**:
여러 개의 작은 업데이트를 한꺼번에 모아서 한 번에 처리하는 기능입니다. 마치 편지를 받을 때마다 우체부가 오는 게 아니라, 하루치 편지를 모아서 한 번에 배달하는 것과 같습니다.

---

### 2. notifyManager.batchCalls
**용도**: 콜백을 감싸서, 호출될 때마다 다음 배치에서 실행되도록 예약

```typescript
type BatchCallsCallback<T extends Array<unknown>> = (...args: T) => void

function batchCalls<T extends Array<unknown>>(
  callback: BatchCallsCallback<T>,
): BatchCallsCallback<T>
```

**쉽게 풀어서**:
원래 함수를 "배치용 함수"로 변환합니다. 변환된 함수를 여러 번 호출해도, 실제로는 다음 배치 타이밍에 모아서 한 번에 실행됩니다.

---

### 3. notifyManager.schedule
**용도**: 함수를 다음 배치에서 실행되도록 예약

```typescript
function schedule(callback: () => void): void
```

**기본 동작**: 
- 기본적으로 `setTimeout(callback, 0)`을 사용
- `setScheduler`로 커스터마이징 가능

**쉽게 풀어서**:
"이 작업은 지금 당장이 아니라 다음 차례에 실행해줘"라고 예약하는 기능입니다.

---

### 4. notifyManager.setNotifyFunction
**용도**: 알림 함수를 오버라이드하여 콜백 실행 방식 커스터마이징

```typescript
// React 테스트 환경에서 사용 예시
import { notifyManager } from '@tanstack/react-query'
import { act } from 'react-dom/test-utils'

notifyManager.setNotifyFunction(act)
```

**쉽게 풀어서**:
- 기본적으로는 그냥 콜백을 바로 실행
- React의 `act()`로 감싸면 테스트 환경에서 안전하게 실행 가능
- "콜백을 어떻게 실행할지" 방식을 바꿀 수 있음

**사용 시나리오**:
- 테스트 환경에서 React 경고 방지
- 특정 프레임워크의 업데이트 사이클과 통합

---

### 5. notifyManager.setBatchNotifyFunction
**용도**: 프레임워크별 배치 업데이트 함수 설정

```typescript
// Solid.js 예시
import { notifyManager } from '@tanstack/query-core'
import { batch } from 'solid-js'

notifyManager.setBatchNotifyFunction(batch)
```

**쉽게 풀어서**:
- 각 프레임워크마다 "여러 업데이트를 묶는 방법"이 다름
- React는 자동으로 배칭하지만, Solid는 `batch()` 함수 사용
- 프레임워크에 맞는 배칭 방법을 TanStack Query에 알려줌

---

### 6. notifyManager.setScheduler
**용도**: 다음 배치가 언제 실행될지 타이밍 설정

```typescript
// 마이크로태스크 큐 사용 (가장 빠름)
notifyManager.setScheduler(queueMicrotask)

// 다음 프레임 렌더링 전
notifyManager.setScheduler(requestAnimationFrame)

// 10ms 후
notifyManager.setScheduler((cb) => setTimeout(cb, 10))
```

**기본값**: `setTimeout(callback, 0)`

**쉽게 풀어서**:
배치를 "언제" 실행할지 결정합니다.
- `queueMicrotask`: 현재 작업이 끝나자마자 바로 (가장 빠름)
- `requestAnimationFrame`: 화면이 다시 그려지기 직전
- `setTimeout`: 지정한 시간 후

**선택 기준**:
- 빠른 반응이 필요하면 → `queueMicrotask`
- 애니메이션과 동기화하려면 → `requestAnimationFrame`
- 디바운싱이 필요하면 → `setTimeout`

---

## 요점 정리

### 왜 배칭이 필요한가?
상태가 변경될 때마다 매번 렌더링하면:
- 성능 저하 발생
- 불필요한 중간 상태가 화면에 나타남

배칭을 사용하면:
- 여러 업데이트를 모아서 한 번에 처리
- 최종 결과만 화면에 반영
- **성능 향상 + 사용자 경험 개선**

### 실전 활용 시나리오

1. **React 테스트 작성 시**
   ```typescript
   notifyManager.setNotifyFunction(act)
   ```
   → 테스트 환경에서 경고 없이 안전하게 상태 업데이트

2. **프레임워크별 최적화**
   ```typescript
   // Solid.js
   notifyManager.setBatchNotifyFunction(batch)
   notifyManager.setScheduler(queueMicrotask)
   ```
   → 각 프레임워크에 최적화된 방식으로 배칭

3. **성능 튜닝**
   ```typescript
   // 애니메이션과 동기화
   notifyManager.setScheduler(requestAnimationFrame)
   ```
   → 화면 갱신 주기에 맞춰 업데이트

### 핵심 개념 정리

| 메서드 | 역할 | 주요 사용처 |
|--------|------|-------------|
| `batch` | 업데이트 묶기 | 내부 최적화 |
| `batchCalls` | 함수 호출 배치화 | 고차 함수 패턴 |
| `schedule` | 실행 예약 | 지연 실행 |
| `setNotifyFunction` | 실행 방식 변경 | 테스트, 통합 |
| `setBatchNotifyFunction` | 배치 함수 설정 | 프레임워크 통합 |
| `setScheduler` | 타이밍 제어 | 성능 튜닝 |

### 기억해야 할 핵심
1. **자동으로 최적화**: 대부분의 경우 기본 설정만으로 충분
2. **필요할 때만 커스터마이징**: 테스트, 특정 프레임워크, 성능 문제가 있을 때만 설정 변경
3. **배칭 = 성능**: 여러 업데이트를 묶어서 처리 → 렌더링 최소화
