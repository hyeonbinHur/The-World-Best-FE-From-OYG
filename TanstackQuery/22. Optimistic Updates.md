# 22. 낙관적 업데이트 (Optimistic Updates)

## 핵심 개념

React Query는 Mutation이 완료되기 전에 UI를 낙관적으로 업데이트하는 두 가지 방법을 제공한다.

1. **UI를 통한 방식**: `useMutation`이 반환하는 `variables`를 활용하여 UI를 직접 업데이트
2. **캐시를 통한 방식**: `onMutate` 옵션에서 캐시를 직접 조작

---

## 방법 1: UI를 통한 낙관적 업데이트

캐시를 직접 건드리지 않으므로 더 간단한 방식이다.

### 기본 사용법

```tsx
const addTodoMutation = useMutation({
  mutationFn: (newTodo: string) => axios.post('/api/data', { text: newTodo }),
  // 쿼리 무효화에서 반환된 Promise를 반드시 return해야
  // refetch가 끝날 때까지 mutation이 pending 상태를 유지한다
  onSettled: () => queryClient.invalidateQueries({ queryKey: ['todos'] }),
})

const { isPending, submittedAt, variables, mutate, isError } = addTodoMutation
```

`addTodoMutation.variables`를 통해 전달한 todo 데이터에 접근할 수 있다. Mutation이 `isPending` 상태인 동안 UI 목록에 임시 항목을 추가할 수 있다.

```tsx
<ul>
  {todoQuery.items.map((todo) => (
    <li key={todo.id}>{todo.text}</li>
  ))}
  {isPending && <li style={{ opacity: 0.5 }}>{variables}</li>}
</ul>
```

- Mutation이 진행 중일 때 반투명한 임시 항목을 렌더링한다.
- 완료되면 임시 항목은 자동으로 사라지고, refetch가 성공하면 정상 항목으로 목록에 나타난다.

### 에러 발생 시 처리

Mutation이 실패하면 임시 항목이 사라지지만, `isError` 상태를 확인하여 계속 표시하고 재시도 버튼을 제공할 수도 있다. `variables`는 에러가 발생해도 초기화되지 않는다.

```tsx
{isError && (
  <li style={{ color: 'red' }}>
    {variables}
    <button onClick={() => mutate(variables)}>Retry</button>
  </li>
)}
```

### Mutation과 Query가 다른 컴포넌트에 있는 경우

`useMutationState` 훅을 사용하면 다른 컴포넌트에서도 Mutation 상태에 접근할 수 있다. `mutationKey`와 함께 사용하면 가장 효과적이다.

```tsx
// 앱 어딘가에서 mutation 정의
const { mutate } = useMutation({
  mutationFn: (newTodo: string) => axios.post('/api/data', { text: newTodo }),
  onSettled: () => queryClient.invalidateQueries({ queryKey: ['todos'] }),
  mutationKey: ['addTodo'],
})

// 다른 컴포넌트에서 mutation 상태에 접근
const variables = useMutationState<string>({
  filters: { mutationKey: ['addTodo'], status: 'pending' },
  select: (mutation) => mutation.state.variables,
})
```

- `variables`는 배열로 반환된다 (동시에 여러 mutation이 실행될 수 있기 때문).
- 고유 키가 필요한 경우 `mutation.state.submittedAt`을 선택하면 동시 낙관적 업데이트 표시도 간단해진다.

## 방법 2: 캐시를 통한 낙관적 업데이트

서버 상태를 낙관적으로 업데이트하면 Mutation이 실패할 가능성이 있다. 대부분의 경우 refetch를 트리거하여 서버 상태로 되돌릴 수 있지만, 서버 문제 등으로 refetch가 불가능한 경우 **롤백(rollback)** 이 필요하다.

`useMutation`의 `onMutate` 핸들러에서 반환한 값은 이후 `onError`와 `onSettled` 핸들러의 마지막 인자로 전달된다. 보통 롤백 데이터를 전달하는 데 사용한다.

### 예시 1: Todo 목록에 새 항목 추가

```tsx
const queryClient = useQueryClient()

useMutation({
  mutationFn: updateTodo,
  // mutate가 호출되면:
  onMutate: async (newTodo, context) => {
    // 진행 중인 refetch를 취소한다
    // (낙관적 업데이트를 덮어쓰지 않도록)
    await context.client.cancelQueries({ queryKey: ['todos'] })

    // 이전 값을 스냅샷으로 저장
    const previousTodos = context.client.getQueryData(['todos'])

    // 새 값으로 낙관적 업데이트
    context.client.setQueryData(['todos'], (old) => [...old, newTodo])

    // 스냅샷 값을 반환
    return { previousTodos }
  },
  // Mutation이 실패하면 onMutate에서 반환한 값으로 롤백
  onError: (err, newTodo, onMutateResult, context) => {
    context.client.setQueryData(['todos'], onMutateResult.previousTodos)
  },
  // 에러든 성공이든 항상 refetch
  onSettled: (data, error, variables, onMutateResult, context) =>
    context.client.invalidateQueries({ queryKey: ['todos'] }),
})
```

### 예시 2: 단일 Todo 업데이트

```tsx
useMutation({
  mutationFn: updateTodo,
  onMutate: async (newTodo, context) => {
    await context.client.cancelQueries({ queryKey: ['todos', newTodo.id] })

    const previousTodo = context.client.getQueryData(['todos', newTodo.id])

    context.client.setQueryData(['todos', newTodo.id], newTodo)

    return { previousTodo, newTodo }
  },
  onError: (err, newTodo, onMutateResult, context) => {
    context.client.setQueryData(
      ['todos', onMutateResult.newTodo.id],
      onMutateResult.previousTodo,
    )
  },
  onSettled: (newTodo, error, variables, onMutateResult, context) =>
    context.client.invalidateQueries({ queryKey: ['todos', newTodo.id] }),
})
```

`onSettled`에서 `onError`와 `onSuccess`를 모두 처리할 수도 있다:

```tsx
useMutation({
  mutationFn: updateTodo,
  // ...
  onSettled: async (newTodo, error, variables, onMutateResult, context) => {
    if (error) {
      // 에러 처리
    }
  },
})
```

## 어떤 방식을 선택할까?

| 기준 | UI를 통한 방식 | 캐시를 통한 방식 |
|------|---------------|-----------------|
| 코드량 | 적다 | 많다 |
| 복잡도 | 간단하다 | 롤백 처리 등이 필요하다 |
| 롤백 처리 | 필요 없다 | 직접 구현해야 한다 |
| 적합한 상황 | 낙관적 결과를 **한 곳**에서만 표시할 때 | 화면의 **여러 곳**에서 업데이트를 반영해야 할 때 |
| 캐시 동기화 | 자동 (refetch 기반) | 자동 (캐시 직접 반영) |

## 요점 정리

| 항목 | 설명 |
|------|------|
| 목적 | Mutation 완료 전에 UI를 먼저 업데이트하여 사용자 경험을 개선 |
| UI 방식 핵심 | `useMutation`의 `variables`와 `isPending`을 활용하여 임시 UI 렌더링 |
| 캐시 방식 핵심 | `onMutate`에서 캐시를 직접 수정하고, 실패 시 `onError`에서 롤백 |
| 캐시 방식 롤백 흐름 | `onMutate`에서 이전 값 스냅샷 저장 -> 낙관적 업데이트 -> 실패 시 `onError`에서 스냅샷으로 복원 |
| 다른 컴포넌트 접근 | `useMutationState` 훅 + `mutationKey`로 다른 컴포넌트에서도 mutation 상태 접근 가능 |
| refetch 취소 | 캐시 방식에서는 `cancelQueries`로 진행 중인 refetch를 취소하여 낙관적 업데이트가 덮어써지는 것을 방지 |
| 선택 기준 | 한 곳에서만 표시 -> UI 방식, 여러 곳에서 반영 필요 -> 캐시 방식 |
