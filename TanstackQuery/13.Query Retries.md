# 13. 쿼리 재시도 (Query Retries)

## 쿼리 재시도란?

`useQuery`의 쿼리 함수가 에러를 던지면(요청 실패), TanStack Query는 **최대 연속 재시도 횟수**에 도달하지 않았다면 자동으로 쿼리를 재시도합니다. 기본값은 **3회**입니다.

재시도는 전역(global) 레벨과 개별 쿼리 레벨 모두에서 설정 가능합니다.

## retry 옵션

| 설정 | 동작 |
|------|------|
| `retry: false` | 재시도 비활성화 |
| `retry: 6` | 실패한 요청을 최대 6번 재시도한 후 최종 에러를 표시 |
| `retry: true` | 실패한 요청을 무한히 재시도 |
| `retry: (failureCount, error) => ...` | 실패 원인에 따라 커스텀 로직으로 재시도 여부 결정. `failureCount`는 첫 번째 재시도 시 0부터 시작 |

서버 환경에서는 서버 렌더링 속도를 최대한 빠르게 하기 위해 retry 기본값이 **0**입니다.

```tsx
import { useQuery } from '@tanstack/react-query'

const result = useQuery({
  queryKey: ['todos', 1],
  queryFn: fetchTodoListPage,
  retry: 10, // 에러 표시 전 최대 10번 재시도
})
```

### error와 failureReason의 관계

마지막 재시도까지는 에러 내용이 `failureReason` 속성에 담깁니다. 위 예제 기준으로 처음 9번의 재시도(총 10번 시도) 동안은 에러가 `failureReason`에 들어가고, 마지막 시도에서도 실패하면 그때서야 `error` 속성에 최종 에러가 들어갑니다.

## Retry Delay (재시도 지연)

TanStack Query에서 재시도는 요청 실패 직후 바로 일어나지 않습니다. 표준적인 방식대로 **지수 백오프(exponential back-off)** 지연이 적용됩니다.

기본 `retryDelay`는 시도마다 **2배씩 증가**(1000ms부터 시작)하되, **최대 30초를 넘지 않습니다**:

```tsx
import {
  QueryCache,
  QueryClient,
  QueryClientProvider,
} from '@tanstack/react-query'

// 전역 설정
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      retryDelay: (attemptIndex) => Math.min(1000 * 2 ** attemptIndex, 30000),
    },
  },
})

function App() {
  return <QueryClientProvider client={queryClient}>...</QueryClientProvider>
}
```

실제 지연 시간 예시:

| 시도 (attemptIndex) | 계산 | 지연 시간 |
|---------------------|------|-----------|
| 0 (첫 번째 재시도) | 1000 * 2^0 | 1초 |
| 1 | 1000 * 2^1 | 2초 |
| 2 | 1000 * 2^2 | 4초 |
| 3 | 1000 * 2^3 | 8초 |
| 4 | 1000 * 2^4 | 16초 |
| 5 이상 | min(계산값, 30000) | 최대 30초 |

### 고정 지연 시간 설정

`retryDelay`에 함수 대신 숫자를 넣으면 항상 동일한 시간만큼 기다립니다:

```tsx
const result = useQuery({
  queryKey: ['todos'],
  queryFn: fetchTodoList,
  retryDelay: 1000, // 재시도 횟수와 무관하게 항상 1초 대기
})
```

## Background Retry Behavior (백그라운드 재시도 동작)

`refetchInterval`과 `refetchIntervalInBackground: true`를 함께 사용할 때, 브라우저 탭이 비활성화되면 **재시도가 일시 중지**됩니다. 재시도는 일반 refetch와 동일한 포커스 동작을 따르기 때문입니다.

백그라운드에서도 지속적으로 재시도가 필요하다면, 기본 재시도를 비활성화하고 커스텀 refetch 전략을 구현하는 방법이 있습니다:

```tsx
const result = useQuery({
  queryKey: ['todos'],
  queryFn: fetchTodos,
  refetchInterval: (query) => {
    // 에러 상태일 때 더 자주 refetch (사실상 재시도 역할)
    return query.state.status === 'error' ? 5000 : 30000
  },
  refetchIntervalInBackground: true,
  retry: false, // 내장 재시도 비활성화
})
```

이 방식은 백그라운드에서도 refetch를 활성화하면서, 에러 상태일 때 더 짧은 간격으로 refetch하여 사실상 재시도 효과를 내는 전략입니다.
